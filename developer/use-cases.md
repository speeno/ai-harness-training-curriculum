# 개발자 프로젝트 Use Case 카탈로그 — 직무 공통 12종

이 문서는 개발자 트랙 교육(AX 워크톤 데이·스프린트·부트캠프)에서 참가자가 "우리 팀 레포"로 바로 치환할 수 있는 **개발 직무 중심 프로젝트 주제 12종**과 **공통 Best Practice 8가지**를 담은 카탈로그입니다.

비개발자 트랙의 직무 공통 카탈로그는 [non-developer/use-cases.md](../non-developer/use-cases.md), 산업별 시나리오는 [materials/industry-use-cases/](../materials/industry-use-cases/README.md)를 사용합니다. 비개발 카탈로그의 주제가 "업무 절차의 스킬화"라면, 이 문서의 주제는 **에이전트가 안전하게 도구를 쓰고 산출물이 검증되는 하네스 구축**에 초점을 둡니다. 비개발 트랙에서 나온 요구사항(스킬 초안·워크플로)을 입력으로 받아 기술 구현으로 연결하는 짝 주제도 표시했습니다.

## 사용 방법

1. 참가자가 아래 매트릭스에서 자기 팀 업무와 가장 가까운 주제를 고릅니다.
2. 해당 주제의 표(Trigger / Input / AI 작업 / 사람 검토 / Output / 검증)를 자기 레포 상황으로 바꿔 적습니다.
3. `SKILL.md` 초안 → 오케스트레이터·Task 설계 → `AGENTS.md` 포인터 → 스모크 검증 순서로 산출물을 만듭니다.
4. 데이(1일)에서는 단일 스킬과 스모크 메모까지, 스프린트(3일)에서는 스킬 패키지와 롤아웃 체크리스트까지, 부트캠프(1주)에서는 CI 연계·런북·로드맵까지 확장합니다.

## 01. PR 리뷰 보조 자동화

팀 리뷰 체크리스트를 스킬로 만들어 PR diff 요약과 리뷰 코멘트 초안을 생성합니다. 전원이 매일 하는 업무라 첫 실습으로 적합하고, "자동 승인 금지" 경계를 배우는 대표 주제입니다.

| 항목 | 내용 |
| --- | --- |
| Trigger | PR 생성 또는 리뷰 요청 시 |
| Input | PR diff, 팀 리뷰 체크리스트, 코딩 컨벤션 문서 |
| AI 작업 | 변경 요약, 체크리스트 항목별 점검 결과와 근거 라인 표시, 리뷰 코멘트 초안 작성 |
| 사람 검토 | 리뷰어가 코멘트 초안을 취사선택해 등록 — 승인·머지는 사람만 수행 |
| Output | 변경 요약과 리뷰 코멘트 초안 |
| 검증 | 샘플 PR 3건에 대한 스모크: 필수 체크 항목 누락 여부, 근거 라인 정확성 |

- **대표 산출물**: 리뷰 체크리스트가 포함된 `SKILL.md`, `AGENTS.md`에 리뷰 스킬 포인터, 스모크 메모
- **리스크**: 스타일 지적에 치우쳐 로직 결함을 놓침, 존재하지 않는 라인 인용
- **Best Practice 포인트**: 스킬에 "지적할 수 없으면 침묵" 규칙과 심각도 라벨(차단/권고/취향)을 정의하면 코멘트 노이즈가 크게 줄어든다.

## 02. 레거시 모듈 문서화와 `AGENTS.md` 정비

문서가 없는 레거시 모듈의 구조·의존성·진입점을 정리하고, 에이전트가 길을 잃지 않도록 `AGENTS.md` 포인터를 만듭니다.

| 항목 | 내용 |
| --- | --- |
| Trigger | 신규 팀원 합류, 레포에 에이전트 자동화 도입 시 |
| Input | 대상 모듈 소스, 빌드·실행 명령, 기존 산발 문서 |
| AI 작업 | 모듈 구조·의존성·진입점 요약, 빌드·테스트 명령 정리, `AGENTS.md` 초안과 금지 경로 후보 표시 |
| 사람 검토 | 모듈 오너가 사실관계와 금지 경로(건드리면 안 되는 디렉터리)를 확정 |
| Output | 모듈 README 초안과 `AGENTS.md` |
| 검증 | 문서만 보고 빌드·테스트가 재현되는지 제3자 확인 |

- **대표 산출물**: 모듈 README, `AGENTS.md`(포인터·금지 경로·로컬 명령), 문서 갱신 절차 메모
- **리스크**: 코드에서 추론한 잘못된 의도 서술, 실제와 다른 명령 기재
- **Best Practice 포인트**: `AGENTS.md`는 짧게 유지하고 세부는 스킬·README로 위임한다. "실행해 보지 않은 명령은 문서에 넣지 않는다"를 검증 기준으로 삼는다.

## 03. 릴리스 노트·변경 요약 자동 생성

커밋·PR 이력에서 릴리스 노트 초안을 만들고, 대상 독자별(개발팀·CS·고객) 버전으로 변환합니다. 비개발 카탈로그 07(콘텐츠 리퍼포징)의 개발자 짝 주제입니다.

| 항목 | 내용 |
| --- | --- |
| Trigger | 릴리스 태그 생성 전 |
| Input | 버전 구간 커밋·PR 목록, 이슈 링크, 릴리스 노트 템플릿 |
| AI 작업 | 변경을 기능·수정·내부 변경으로 분류, 사용자 영향 중심으로 재서술, 독자별 버전 생성 |
| 사람 검토 | 릴리스 담당자가 고객 공지 문구와 브레이킹 체인지 표기를 확인 |
| Output | 독자별 릴리스 노트 초안 |
| 검증 | 브레이킹 체인지·마이그레이션 필요 항목이 누락 없이 표기됐는지 대조 |

- **대표 산출물**: 분류 기준과 템플릿이 포함된 `SKILL.md`, 태그 → 초안 → 검토 → 배포 워크플로
- **리스크**: 브레이킹 체인지 누락, 내부용 표현이 고객 공지에 유출
- **Best Practice 포인트**: PR 라벨·컨벤셔널 커밋 등 기존 메타데이터를 분류 근거로 강제하면 재현성이 생긴다. 메타데이터가 없는 커밋은 "분류 불가"로 표시하게 한다.

## 04. 테스트 커버리지 보강 스킬

커버리지 리포트에서 미커버 경로를 찾아 팀 컨벤션에 맞는 테스트 코드를 생성합니다. "생성된 테스트를 어떻게 믿을 것인가"라는 검증 문제를 정면으로 다룹니다.

| 항목 | 내용 |
| --- | --- |
| Trigger | 커버리지 기준 미달 모듈 지정 시 |
| Input | 대상 코드, 커버리지 리포트, 기존 테스트(컨벤션 참조용), 테스트 러너 설정 |
| AI 작업 | 미커버 분기 목록화, 우선순위 제안, 팀 컨벤션에 맞는 테스트 코드 생성, 실행 결과 확인 |
| 사람 검토 | 개발자가 테스트의 의미(잘못된 동작을 고정하는 테스트가 아닌지) 리뷰 |
| Output | 테스트 코드 PR과 커버리지 변화 요약 |
| 검증 | 생성 테스트 전체 통과 + 변이(mutation) 확인 표본: 일부러 버그를 넣었을 때 실패하는가 |

- **대표 산출물**: 테스트 컨벤션 `SKILL.md`, 미커버 탐지 → 생성 → 실행 → 리뷰 워크플로
- **리스크**: 현재 버그를 정답으로 고정하는 테스트, 커버리지 숫자만 올리는 무의미한 테스트
- **Best Practice 포인트**: "통과하는 테스트"가 아니라 "버그를 넣으면 실패하는 테스트"를 통과 기준으로 정의한다. 스킬에 단언(assertion) 없는 테스트 금지를 명시한다.

## 05. 로그·에러 트리아지 브리프

에러 로그와 알림을 클러스터링해 신규·급증·기지(旣知) 이슈를 구분하고 이슈 트래커 초안을 만듭니다.

| 항목 | 내용 |
| --- | --- |
| Trigger | 매일 오전, 또는 에러율 임계치 초과 시 |
| Input | 에러 로그 내보내기(마스킹 처리), 기존 이슈 목록, 최근 배포 이력 |
| AI 작업 | 유사 에러 클러스터링, 신규·급증·기지 분류, 최근 배포와의 상관 표시, 이슈 초안 작성 |
| 사람 검토 | 담당 개발자가 우선순위와 담당 지정 — 이슈 등록은 사람 승인 후 |
| Output | 트리아지 브리프와 이슈 초안 |
| 검증 | 기지 이슈 오분류율 표본 확인, 마스킹 누락 검사 |

- **대표 산출물**: 클러스터링·분류 기준 `SKILL.md`, 로그 마스킹 전처리 단계가 포함된 워크플로
- **리스크**: 로그 내 PII·시크릿 노출, 서로 다른 원인을 하나의 이슈로 병합
- **Best Practice 포인트**: 마스킹을 AI 입력 전 파이프라인 단계로 고정한다(스킬 규칙이 아니라 전처리로). "배포와의 상관은 가설로만 표기, 원인 단정 금지"를 출력 규칙에 넣는다.

## 06. 표준 스캐폴딩 스킬화

새 서비스·모듈·엔드포인트 생성 시 반복되는 보일러플레이트를 사내 표준이 반영된 스캐폴딩 스킬로 만듭니다.

| 항목 | 내용 |
| --- | --- |
| Trigger | 새 모듈·서비스 생성 요청 시 |
| Input | 사내 표준 템플릿, 네이밍 규칙, 디렉터리 컨벤션, 참조할 모범 모듈 1개 |
| AI 작업 | 표준 구조로 스캐폴딩 생성, 네이밍·설정 파일 자동 반영, 생성 후 체크리스트 출력 |
| 사람 검토 | 개발자가 생성 결과를 빌드·실행해 확인 후 커밋 |
| Output | 표준 구조의 신규 모듈 스캐폴딩 |
| 검증 | 생성 직후 빌드·린트·기본 테스트 통과를 스모크로 자동 확인 |

- **대표 산출물**: 스캐폴딩 `SKILL.md`(모범 모듈 참조 포함), 생성 → 스모크 → 커밋 워크플로
- **리스크**: 구버전 템플릿 반영, 표준 개정 시 스킬 미갱신으로 표준 이탈 확산
- **Best Practice 포인트**: 템플릿 원본을 스킬에 복사하지 말고 레포 내 모범 모듈을 참조하게 하면, 표준이 바뀔 때 스킬이 자동으로 따라간다.

## 07. 의존성 업그레이드 브리프

취약점 공지나 메이저 버전 릴리스 시 영향 범위를 정리하고 업그레이드 PR 준비 자료를 만듭니다.

| 항목 | 내용 |
| --- | --- |
| Trigger | 보안 공지 수신, 분기별 의존성 점검 시 |
| Input | 의존성 목록(lockfile), 대상 패키지 체인지로그, 사용처 검색 결과 |
| AI 작업 | 사용처와 브레이킹 체인지 대조, 영향 파일 목록화, 업그레이드 순서와 테스트 범위 제안 |
| 사람 검토 | 담당 개발자가 영향 분석을 검증하고 업그레이드 실행 여부 결정 |
| Output | 업그레이드 영향 브리프와 작업 체크리스트 |
| 검증 | 영향 파일 목록을 실제 grep·컴파일 결과와 표본 대조 |

- **대표 산출물**: 영향 분석 `SKILL.md`, 공지 → 분석 → 결정 → 업그레이드 → 회귀 확인 워크플로
- **리스크**: 간접 의존성 영향 누락, 체인지로그에 없는 동작 변화
- **Best Practice 포인트**: "체인지로그 기반 분석은 참고, 최종 판정은 테스트 통과"로 신뢰 순서를 명시한다. 취약점 대응은 별도 긴급 경로(SLA)를 워크플로에 정의한다.

## 08. 사내 API·코드베이스 Q&A 스킬

내부 API 문서와 코드를 참조 자료로 연결해 "이 기능 어디서 처리해요?" 류 반복 질문에 근거 링크와 함께 답하는 스킬을 만듭니다. 비개발 카탈로그 06(규정 Q&A)의 개발자 짝 주제입니다.

| 항목 | 내용 |
| --- | --- |
| Trigger | 온보딩·타 팀 문의 채널의 반복 질문 |
| Input | API 문서, 아키텍처 문서, 코드 검색 결과 |
| AI 작업 | 질문에 해당하는 코드 위치·문서를 찾아 파일 경로와 라인 인용으로 답변 초안 작성, 근거 없으면 "확인 필요" 표시 |
| 사람 검토 | 모듈 오너가 답변 정확성 확인 — 특히 "지금도 그런가" 최신성 |
| Output | 근거 경로가 인용된 답변, 자주 묻는 질문 축적 문서 |
| 검증 | 인용 경로·라인이 실제 존재하고 내용과 일치하는지 표본 확인 |

- **대표 산출물**: 참조 자료 연결 규칙이 있는 Q&A `SKILL.md`, 답변 축적 → FAQ 문서화 루프
- **리스크**: 삭제된 코드 인용, 문서와 코드 불일치 시 문서를 정답으로 오인
- **Best Practice 포인트**: "코드와 문서가 다르면 코드가 정답, 단 불일치 자체를 리포트"를 규칙으로 넣으면 Q&A가 문서 정비 신호 역할까지 한다.

## 09. 데이터 파이프라인 검증 리포트

파이프라인 실행 후 스키마 변화, 행 수 급변, 널 비율 이상을 점검해 데이터 품질 브리프를 만듭니다. 비개발 카탈로그 09(예산 점검)의 개발자 짝 주제입니다.

| 항목 | 내용 |
| --- | --- |
| Trigger | 일 배치 완료 후, 스키마 변경 배포 후 |
| Input | 실행 로그, 테이블 통계(행 수·널 비율·신규 값), 스키마 정의, 전일 리포트 |
| AI 작업 | 전일 대비 이상치 표시, 스키마 변경과 다운스트림 영향 정리, 담당자별 확인 요청 초안 |
| 사람 검토 | 데이터 엔지니어가 이상 판정 기준과 수치를 원본 쿼리로 대조 |
| Output | 데이터 품질 브리프와 확인 요청 목록 |
| 검증 | 이상 판정 수치를 원본과 표본 대조 — 수치 산출물은 대조 없이는 미완성 |

- **대표 산출물**: 이상 기준 정의 `SKILL.md`, 배치 → 점검 → 브리프 → 확인 회신 워크플로
- **리스크**: 계산 오류를 그대로 전달, 정상 계절성 변동을 이상으로 오탐
- **Best Practice 포인트**: 판정 기준(임계값)은 AI가 아니라 스킬에 사람이 명시한 수치를 쓴다. 기준 변경은 스킬 버전 업데이트로만 가능하게 한다.

## 10. 온콜 인시던트 초동 대응 보조

알림 발생 시 런북 기반 진단 체크리스트를 제시하고, 대응 타임라인을 기록해 포스트모템 초안까지 만듭니다.

| 항목 | 내용 |
| --- | --- |
| Trigger | 온콜 알림 발생 시, 인시던트 종료 후 |
| Input | 알림 내용, 런북, 최근 배포 이력, 대응 채널 로그(타임라인용) |
| AI 작업 | 해당 런북 절차 제시, 확인 항목 체크리스트화, 채널 로그에서 타임라인 정리, 포스트모템 초안 작성 |
| 사람 검토 | 온콜 엔지니어가 모든 조치를 직접 실행 — AI는 조치 실행 금지, 안내와 기록만 |
| Output | 초동 대응 체크리스트, 타임라인, 포스트모템 초안 |
| 검증 | 모의 인시던트(게임데이)로 런북 안내 정확성 리허설 |

- **대표 산출물**: 런북 참조 `SKILL.md`, 알림 → 안내 → 기록 → 포스트모템 워크플로, 조치 실행 금지 경계 정의
- **리스크**: 잘못된 런북 안내로 장애 악화, 타임라인 순서 오류로 포스트모템 왜곡
- **Best Practice 포인트**: "읽기 전용" 경계를 기술적으로 강제한다(프로덕션 변경 권한 없는 계정). 포스트모템 초안에는 "AI 생성 초안, 사실 검증 전" 워터마크를 자동 삽입한다.

## 11. 대규모 마이그레이션 파이프라인

API 교체, 프레임워크 업그레이드 같은 수백 파일 규모 변환을 탐지 → 변환 → 검증 파이프라인으로 설계합니다. Task·서브에이전트 병렬 실행과 멱등성을 다루는 대표 심화 주제입니다.

| 항목 | 내용 |
| --- | --- |
| Trigger | 마이그레이션 프로젝트 착수 시 |
| Input | 변환 규칙 정의, 대상 탐지 쿼리, 변환 전후 예시 쌍, 테스트 스위트 |
| AI 작업 | 대상 파일 목록화, 파일 단위 병렬 변환(서브에이전트), 변환 불가 케이스 별도 분류, 배치별 테스트 실행 |
| 사람 검토 | 배치 단위 PR 리뷰 — 전체 일괄 머지 금지, 변환 불가 케이스는 사람이 직접 처리 |
| Output | 배치별 변환 PR, 변환 불가 목록, 진행 대시보드 |
| 검증 | 배치마다 테스트 통과 + 표본 육안 대조, 실패 배치는 롤백 후 규칙 수정 |

- **대표 산출물**: 변환 규칙 `SKILL.md`, 오케스트레이터(탐지·분배·취합), Task 설계 문서(병렬 단위·멱등성·재시도), 배치 롤아웃 체크리스트
- **리스크**: 문맥 의존 코드의 기계적 오변환, 중간 실패 시 반쯤 변환된 상태 방치
- **Best Practice 포인트**: 배치 크기를 작게 시작해 규칙을 수렴시킨 뒤 확대한다. 같은 배치를 재실행해도 결과가 같도록(멱등) 산출물 단위를 설계하고, "변환 불가로 분류하는 것도 성공"임을 규칙에 명시한다.

## 12. 에이전트 스모크·회귀 검증 하네스 구축

팀의 에이전트 산출물(스킬 실행 결과)을 자동 검증하는 스모크·회귀 체계를 만들고 CI에 연결합니다. 다른 모든 주제의 품질을 받쳐 주는 기반 주제로, 부트캠프의 종합 프로젝트에 적합합니다.

| 항목 | 내용 |
| --- | --- |
| Trigger | 스킬 신규 등록·변경 시(CI), 정기 회귀 실행 시 |
| Input | 검증 대상 스킬 목록, 입력 픽스처, 기대 출력 기준(필수 포함 항목·금지 표현·형식) |
| AI 작업 | 픽스처 기반 스킬 실행, 기준 대비 통과·실패 판정, 실패 원인 요약 리포트 |
| 사람 검토 | 플랫폼 담당자가 실패 판정 검토, 기준 자체의 개정 여부 결정 |
| Output | 스모크·회귀 리포트, CI 잡 설정, 실패 시 차단 규칙 |
| 검증 | 알려진 불량 출력을 넣었을 때 검증이 실제로 실패하는지(검증기의 검증) |

- **대표 산출물**: 검증 기준 정의 문서, 회귀 픽스처 세트, CI 잡 설정 초안, 스킬 등록 절차(검증 통과 없이는 등록 불가)
- **리스크**: 형식만 검사해 내용 오류 통과, 픽스처가 낡아 회귀 검증이 형식화
- **Best Practice 포인트**: 실제 운영에서 발견된 실패 사례를 회귀 픽스처로 계속 편입하는 절차를 만들면 검증 체계가 살아 있는 자산이 된다. 검증 기준은 비개발 트랙의 검증 체크리스트와 같은 원천을 공유한다.

## 공통 Best Practice 8가지

모든 주제에 공통 적용되는 원칙입니다. 산출물 리뷰와 롤아웃 체크리스트의 기본 항목으로 사용합니다.

1. **에이전트 산출물에 사람 게이트** — 코드는 PR 리뷰, 외부 발송물은 승인 후 전달. 자동 머지·자동 배포·자동 발송은 파일럿 범위에서 금지한다.
2. **스킬은 코드처럼 관리** — `SKILL.md`는 레포에서 버전 관리하고 변경은 PR로 리뷰한다. 버전과 변경 사유를 기록해 "누가 언제 왜 바꿨는지" 추적 가능하게 한다.
3. **검증 기준을 스킬보다 먼저** — 기대 출력 예시, 필수 포함 항목, 금지 표현, 기대 로그 키워드를 먼저 정의한다. 스모크 없는 스킬은 등록하지 않는다.
4. **`AGENTS.md`는 짧게, 금지 경로는 명시적으로** — 포인터와 로컬 명령, 건드리면 안 되는 경로만 담고 세부는 스킬로 위임한다. 길어진 `AGENTS.md`는 읽히지 않는다.
5. **Task 분해는 바운디드 컨텍스트로** — 서브에이전트에는 필요한 최소 컨텍스트와 명확한 완료 기준만 전달한다. 병렬 단위는 서로 파일이 겹치지 않게 설계한다.
6. **멱등성과 부분 실패 복구** — 같은 입력 재실행 시 같은 결과가 나오게 산출물 단위를 설계하고, 중간 실패 시 어디서부터 다시 하는지 정의한다.
7. **시크릿·PII는 입력 전 차단** — 마스킹은 스킬 규칙이 아니라 전처리 파이프라인으로 강제한다. 외부에서 온 텍스트(이슈·로그·문서)는 프롬프트 인젝션 가능성을 전제로 도구 권한을 최소화한다.
8. **롤백 조건을 롤아웃보다 먼저** — 중단 기준, 롤백 절차, 모니터링 항목을 정한 뒤 파일럿을 시작한다. "언제 멈출지" 없는 자동화는 배포하지 않는다.
