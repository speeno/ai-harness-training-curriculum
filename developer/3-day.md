# 개발자 트랙 · 3일 과정

**형태:** 3일 연속 또는 주 1회 × 3주(각일 6~7시간 + 일일 과제)

---

## 1. 교육 목표 (회사 니즈: 업무 자동화·생산성)

- 팀 표준에 맞춘 **에이전트 정의·스킬 패키지** 설계 역량을 확보한다.
- **오케스트레이터 스킬**로 다단계 워크플로를 조립하고, **Task·서브에이전트**로 병렬·위임을 분리한다.
- **`AGENTS.md`와 규칙 파일**을 통해 온보딩 비용을 줄인다.
- **검증 전략**(단위·통합·에이전트 스모크)을 선택하고 최소 파이프라인을 제안한다.

---

## 2. 대상 수강자

- 서비스·플랫폼·내부 도구를 유지보수하는 **시니어 이상 또는 중급+ 멘토링 가능** 인원
- Git 브랜치 전략과 코드 리뷰 문화에 익숙한 팀

---

| 시간 | 내용 | 형태 |

## 4. 강의 내용 상세 (모듈/토픽)

- **스킬 패키지:** 디렉터리 레이아웃, 공유 유틸, 브레이킹 체인지 공지 방법
- **오케스트레이터:** 부분 실패 시 보상 트랜잭션 개념, 사용자에게 되돌릴 메시지
- **Task 모델:** 입력 스키마(느슨할수록 문서화 부담 증가), 출력 스키마 검증
- **서브에이전트:** 역할 한정, 컨텍스트 요약 전달, 부모-자식 재진입 방지
- **AGENTS.md:** 여러 스킬 중 기본 경로, `README`와 중복 방지
- **검증:** 결정적 단계는 코드 테스트, 비결정적 단계는 샘플 기반 스모크
- **운영:** 로그 필드 표준, 알림 임계값, 롤백 트리거

---

## 5. 진행 방법 (강사 진행 팁, 실습 순서, 소그룹/페어, 준비물)

- **강사 팁:** 2일차에 **의도적 버그**(재시도 누락 등)를 넣은 예제로 디버깅 랩을 진행한다.
- **실습 순서:** 동일 시나리오를 Day1 단일 스킬 → Day2 Task 분해 → Day3 검증 추가로 진화시킨다.
- **소그룹:** 오케스트레이터 설계는 2인 1조로 **시퀀스 다이어그램**부터 그리게 한다.
- **준비물:** 개발자 1일 과정 준비물 + 테스트 러너 설정, 샘플 API 모킹 수단(예: WireMock/로컬 스텁).

---

## 6. 하네스·스킬 연계 (개발자 관점)

| 구분 | 심화 포인트 |
|------|-------------|
| **에이전트 정의** | 역할별 시스템 프롬프트 최소화, 스킬로 밀어넣기 |
| **SKILL.md** | 변경 이력, 호환 에이전트 버전 범위 |
| **오케스트레이터 스킬** | 서브 스킬 실패 매핑표 |
| **Task·서브에이전트** | 동시성 한도, 데드락 가능 구조 회피 |
| **AGENTS.md** | 신규 기여자가 첫 PR 전 읽을 순서 |
| **검증** | 회귀 스위트에 "에이전트 한 번 실행" 단계 포함 여부 결정 |

비개발 트랙에서 내려온 요구사항은 **입력 데이터 fixture**로 저장해 테스트 자산화한다.

---

## 7. 산출물 (수강생 결과물)

1. **스킬 패키지:** 최소 2개 SKILL.md + 오케스트레이터 1개.
2. **Task 설계 문서:** 서브에이전트 목록, 실패 모드, 재시도 정책.
3. **AGENTS.md 업데이트:** 포인터·금지 경로·로컬 실행 명령.
4. **테스트:** 단위 또는 스모크 스크립트 1개 이상, README에 실행법.
5. **롤아웃 체크리스트:** 모니터링 항목·롤백 조건.
