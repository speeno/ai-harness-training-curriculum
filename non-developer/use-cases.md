# 비개발자 프로젝트 Use Case 카탈로그 — 직무 공통 12종

이 문서는 비개발자 트랙 교육(AX 워크톤 데이·스프린트·부트캠프)에서 참가자가 "내 업무"로 바로 치환할 수 있는 **직무(기능) 중심 프로젝트 주제 12종**과 **공통 Best Practice 8가지**를 담은 카탈로그입니다.

산업별 시나리오는 [materials/industry-use-cases/](../materials/industry-use-cases/README.md)의 7종(유통·회계·제조·물류·헬스케어·교육 프랜차이즈·생활용품)을 사용하고, 이 문서는 **산업과 무관하게 어느 조직에나 존재하는 공통 업무**를 다룹니다. 교육 설계 시 두 축을 조합합니다: 산업 축으로 회사 상황을 맞추고, 직무 축으로 참가자 개인의 파일럿 주제를 고릅니다. 개발자 트랙용 카탈로그는 [developer/use-cases.md](../developer/use-cases.md)를 사용합니다.

## 사용 방법

1. 참가자가 아래 매트릭스에서 자기 업무와 가장 가까운 주제를 고릅니다.
2. 해당 주제의 표(Trigger / Input / AI 작업 / 사람 검토 / Output)를 자기 팀 상황으로 바꿔 적습니다.
3. 스킬 초안 → 워크플로 → `_workspace` 규약 → 검증 체크리스트 순서로 산출물을 만듭니다.
4. 난이도와 추천 과정에 따라 데이(1일)에서는 초안까지, 스프린트(3일)에서는 파일럿 설계까지, 부트캠프(1주)에서는 운영 모델까지 확장합니다.

## 01. 주간 업무 보고 자동화

팀원 개인 메모, 협업 도구 현황, 지난주 보고를 모아 주간 보고 초안을 만드는 가장 보편적인 입문 주제입니다. 참가자 전원이 자기 사례를 갖고 있어 첫 실습으로 적합합니다.

| 항목 | 내용 |
| --- | --- |
| Trigger | 매주 금요일 오전, 팀원별 업무 메모 취합 완료 |
| Input | 개인 업무 메모, 협업 보드 현황 내보내기, 지난주 보고서 |
| AI 작업 | 완료·진행·지연 항목 분류, 지난주 대비 변경점 표시, 보고 양식에 맞춰 초안 작성 |
| 사람 검토 | 팀장이 지연 사유 표현과 대외 공유 범위를 확인 |
| Output | 팀 주간 보고 초안 1건 |

- **대표 산출물**: 보고 양식이 포함된 스킬 초안, 취합 → 초안 → 검토 → 발송 워크플로, `_workspace` 폴더 규약
- **리스크**: 지연 사유를 단정적으로 서술, 개인 성과 비교로 오해될 표현
- **Best Practice 포인트**: 보고 양식 자체를 스킬의 예시 출력으로 넣으면 품질 편차가 크게 줄어든다. "작성"이 아니라 "초안 작성 후 사람 검토"로 워크플로를 명시한다.

## 02. 회의록 → 액션 아이템 트래킹

회의 메모에서 결정 사항, 액션 아이템, 담당자, 기한을 추출하고, 다음 회의 전에 미완료 항목 리마인드 브리프를 만듭니다.

| 항목 | 내용 |
| --- | --- |
| Trigger | 회의 종료 후 메모 업로드, 그리고 다음 회의 D-1 |
| Input | 회의 메모 또는 녹취 정리본, 직전 회의 액션 아이템 목록 |
| AI 작업 | 결정·액션·보류 구분, 담당자·기한 추출, 미완료 항목 리마인드 문안 작성 |
| 사람 검토 | 회의 주재자가 담당자·기한 오추출 여부를 확인 |
| Output | 액션 아이템 표와 리마인드 브리프 |

- **대표 산출물**: 액션 추출 기준(무엇이 "결정"인가)을 담은 스킬, 회의 전후 2단계 워크플로
- **리스크**: 발언을 결정으로 오인, 담당자 잘못 지정
- **Best Practice 포인트**: "명시적으로 합의된 것만 결정으로 분류, 모호하면 '확인 필요'로 표시" 규칙을 스킬 금지 사항에 넣는다.

## 03. 고객 문의 1차 분류와 답변 초안

이메일·게시판 문의를 유형별로 분류하고 FAQ 기반 답변 초안을 만듭니다. 산업 use case 04(물류 클레임), 05(헬스케어 민원)의 직무 공통 버전입니다.

| 항목 | 내용 |
| --- | --- |
| Trigger | 매일 오전·오후 문의 채널 확인 시점 |
| Input | 신규 문의 목록, FAQ 문서, 답변 톤 가이드, 에스컬레이션 기준 |
| AI 작업 | 문의를 유형·긴급도로 분류, FAQ 매칭 시 답변 초안 작성, 기준 초과 건은 에스컬레이션 표시 |
| 사람 검토 | 담당자가 초안 검토 후 발송 — 자동 발송 금지 |
| Output | 분류표, 답변 초안, 에스컬레이션 목록 |

- **대표 산출물**: FAQ·톤 가이드를 참조 자료로 연결한 스킬, 분류 → 초안 → 사람 발송 워크플로
- **리스크**: 고객 개인정보 노출, 보상·환불을 임의로 약속하는 표현
- **Best Practice 포인트**: "금액·보상·법적 판단이 포함된 답변은 초안 단계에서 [검토 필수] 태그"를 스킬 규칙으로 강제한다.

## 04. RFP·제안 요청 분석과 대응 체크리스트

고객사 RFP나 입찰 공고에서 요구사항을 추출해 대응 가능·불가·확인 필요로 분류하고, 제안 준비 체크리스트를 만듭니다.

| 항목 | 내용 |
| --- | --- |
| Trigger | RFP 접수 시 |
| Input | RFP 원문, 자사 서비스·레퍼런스 요약 문서, 과거 제안서 목차 |
| AI 작업 | 요구사항 항목화, 배점·필수 여부 표시, 자사 역량 대비 가능/불가/확인 필요 분류, 담당 부서별 확인 질문 생성 |
| 사람 검토 | 영업 대표가 분류 결과와 전략적 판단(전략적 수용 여부)을 결정 |
| Output | 요구사항 대응표, 부서별 확인 요청 목록, 제안 준비 체크리스트 |

- **대표 산출물**: 요구사항 추출·분류 스킬, 영업 → 유관 부서 → 취합 워크플로
- **리스크**: 요구사항 누락(특히 별첨·표 안의 조건), 대응 가능 여부를 과신
- **Best Practice 포인트**: "가능/불가 판단은 AI가 아니라 부서 확인 후 확정"으로 역할을 나눈다. AI의 역할은 빠짐없는 항목화와 질문 생성까지다.

## 05. 채용 서류 스크리닝 보조

JD 요건 대비 지원 서류를 정리해 면접관이 빠르게 검토할 수 있는 요약표와 면접 질문 초안을 만듭니다. 공정성 이슈가 있어 사람 판단 경계를 배우기 좋은 주제입니다.

| 항목 | 내용 |
| --- | --- |
| Trigger | 서류 접수 마감 후 |
| Input | JD(자격 요건·우대 사항), 비식별 처리된 지원 서류 |
| AI 작업 | 요건별 충족 근거를 서류 원문 인용으로 정리, 확인 필요 항목 표시, 경력 기반 면접 질문 초안 |
| 사람 검토 | 채용 담당자와 면접관이 판단 — AI는 합격·불합격을 제안하지 않는다 |
| Output | 요건 매칭표(근거 인용 포함), 면접 질문 초안 |

- **대표 산출물**: "판단 금지, 근거 정리만" 경계가 명시된 스킬, 비식별 처리 → 정리 → 면접관 전달 워크플로
- **리스크**: 성별·연령·출신 등 편향 요소 반영, 개인정보 처리 위반
- **Best Practice 포인트**: 입력 전 비식별 처리를 워크플로의 첫 단계로 고정하고, 스킬 금지 사항에 "점수화·서열화·합불 표현 금지"를 명시한다.

## 06. 사내 규정·정책 Q&A 스킬

경비 처리, 휴가, 보안 규정 등 반복 문의에 대해 규정 원문을 근거로 인용하는 답변 초안을 만듭니다. 참조 자료 관리와 근거 인용을 배우는 대표 주제입니다.

| 항목 | 내용 |
| --- | --- |
| Trigger | 규정 관련 문의 접수 시 |
| Input | 사내 규정·정책 문서(조항 번호 포함), 자주 묻는 질문 기록 |
| AI 작업 | 질문에 해당하는 조항을 찾아 인용과 함께 답변 초안 작성, 규정에 없는 사안은 "규정 없음, 담당자 문의"로 표시 |
| 사람 검토 | 총무·컴플라이언스 담당자가 인용 정확성 확인 후 회신 |
| Output | 근거 조항이 인용된 답변 초안 |

- **대표 산출물**: 규정 문서를 참조 자료로 연결한 Q&A 스킬, 규정 개정 시 스킬 업데이트 절차
- **리스크**: 개정 전 규정 인용, 규정에 없는 내용을 그럴듯하게 생성
- **Best Practice 포인트**: "모든 답변에 조항 번호 인용 필수, 인용할 조항이 없으면 답변 생성 금지"가 환각을 막는 핵심 규칙이다. 규정 개정일을 스킬에 버전으로 기록한다.

## 07. 마케팅 콘텐츠 리퍼포징

블로그 글 1건을 뉴스레터, SNS 채널별 게시물, 보도자료 초안으로 변환합니다. 톤앤매너 가이드의 스킬화를 배우는 주제입니다.

| 항목 | 내용 |
| --- | --- |
| Trigger | 원본 콘텐츠 발행 확정 시 |
| Input | 원본 콘텐츠, 채널별 톤앤매너 가이드, 채널별 형식 제약(글자 수 등), 금지 표현 목록 |
| AI 작업 | 채널별 형식·톤에 맞춘 변환 초안 작성, 핵심 메시지 유지 여부 자체 점검표 첨부 |
| 사람 검토 | 마케터가 브랜드 표현과 사실관계 확인 후 발행 |
| Output | 채널별 콘텐츠 초안 세트 |

- **대표 산출물**: 채널별 하위 규칙을 가진 콘텐츠 변환 스킬, 원본 → 변환 → 검수 → 발행 워크플로
- **리스크**: 과장·단정 표현 생성, 채널 규정(광고 표기 등) 위반
- **Best Practice 포인트**: 채널마다 "좋은 예시 1건, 나쁜 예시 1건"을 스킬에 넣으면 톤 일관성이 눈에 띄게 좋아진다.

## 08. 경쟁사·시장 동향 주간 브리프

지정한 출처(뉴스, 공시, 보고서)의 스크랩을 요약해 팀 관점의 시사점 초안이 달린 주간 브리프를 만듭니다.

| 항목 | 내용 |
| --- | --- |
| Trigger | 매주 월요일 오전, 지난주 스크랩 취합 완료 |
| Input | 출처별 스크랩(링크·본문), 관심 키워드 목록, 지난 브리프 |
| AI 작업 | 항목별 3줄 요약과 출처 표기, 관심 키워드 관련성 표시, 지난주와 이어지는 이슈 연결 |
| 사람 검토 | 기획 담당자가 시사점을 직접 작성·보강 — 요약은 AI, 해석은 사람 |
| Output | 출처가 표기된 주간 동향 브리프 |

- **대표 산출물**: 출처 표기 규칙이 포함된 요약 스킬, 스크랩 → 요약 → 시사점 → 배포 워크플로
- **리스크**: 출처 없는 내용 혼입, 오래된 기사를 최신으로 오인
- **Best Practice 포인트**: "요약문 모든 항목에 원문 링크와 날짜 필수, 링크 없는 항목은 삭제"를 검증 체크리스트 1번에 둔다.

## 09. 예산 집행 점검 브리프

부서 예산 대비 집행 내역을 점검해 이상 항목(초과, 급증, 미집행)을 표시하고 소명 요청 초안을 만듭니다.

| 항목 | 내용 |
| --- | --- |
| Trigger | 매월 첫 영업일, 전월 집행 내역 확정 후 |
| Input | 예산 계획표, 집행 내역 내보내기(비식별 샘플), 전월 브리프 |
| AI 작업 | 항목별 집행률 계산, 기준(예: 계획 대비 ±20%) 초과 항목 표시, 부서별 소명 요청 문안 초안 |
| 사람 검토 | 재무 담당자가 이상 판정 기준과 수치를 검증 — 수치는 원본 대조 필수 |
| Output | 집행 점검표와 소명 요청 초안 |

- **대표 산출물**: 이상 기준이 명시된 점검 스킬, 점검 → 검증 → 소명 요청 → 회신 취합 워크플로
- **리스크**: 계산 오류를 그대로 전달, 이상 아닌 항목에 소명 요청(부서 신뢰 저하)
- **Best Practice 포인트**: 수치가 들어가는 산출물은 "AI 계산 결과를 원본과 표본 대조"를 검증 체크리스트에 반드시 넣는다. 숫자 검증 없는 재무 자동화는 파일럿에서 제외한다.

## 10. 계약서 검토 준비 패키지

표준 계약서 대비 상대방 수정안의 차이점을 정리해 법무 검토 요청서를 준비합니다. AI가 법률 판단을 하지 않는 경계 설계가 핵심입니다.

| 항목 | 내용 |
| --- | --- |
| Trigger | 상대방 수정 계약안 수신 시 |
| Input | 자사 표준 계약서, 상대방 수정안, 과거 유사 계약의 법무 코멘트(비식별) |
| AI 작업 | 조항별 차이점 표(원문 대조 인용), 과거 유사 쟁점 표시, 법무 검토 요청서 초안 — 유불리 판단 금지 |
| 사람 검토 | 현업 담당자가 사실관계 확인 후 법무 부서에 전달, 판단은 법무가 수행 |
| Output | 조항 대조표와 법무 검토 요청서 초안 |

- **대표 산출물**: "차이점 정리까지만" 경계가 명시된 스킬, 현업 → 법무 → 회신 반영 워크플로
- **리스크**: 차이점 누락으로 검토 범위 축소, AI 표현을 법률 자문으로 오인
- **Best Practice 포인트**: 산출물 머리말에 "본 문서는 법률 자문이 아니며 법무 검토 전 초안임"을 자동 삽입하도록 스킬 출력 양식에 고정한다.

## 11. 온보딩 가이드 생성·업데이트

팀 위키와 기존 문서에서 신규 입사자 첫 2주 온보딩 가이드를 생성하고, 조직 변경 시 최신화합니다. 문서가 흩어진 팀의 표준화 진입점으로 좋습니다.

| 항목 | 내용 |
| --- | --- |
| Trigger | 신규 입사 확정 시, 분기별 정기 점검 시 |
| Input | 팀 위키·매뉴얼, 조직도, 계정·권한 신청 절차, 기존 온보딩 후기 |
| AI 작업 | 역할별 첫 2주 체크리스트 생성, 깨진 링크·최신 아닌 정보 표시, 담당자 확인 질문 목록 작성 |
| 사람 검토 | 팀 리더가 순서와 담당자 지정 확인, 버디가 실사용 후 피드백 |
| Output | 온보딩 가이드와 체크리스트, 문서 정비 목록 |

- **대표 산출물**: 온보딩 문서 구조 스킬, 입사 확정 → 생성 → 검토 → 입사 후 피드백 반영 워크플로
- **리스크**: 폐기된 절차 안내, 권한 신청 누락
- **Best Practice 포인트**: 신규 입사자의 실사용 피드백을 다음 버전 입력으로 되돌리는 개선 루프를 워크플로에 포함하면, 일회성 문서가 아니라 관리되는 업무 자산이 된다.

## 12. 행사·웨비나 운영 워크플로

행사 기획부터 사후 팔로업까지 전 단계를 하나의 오케스트레이션으로 설계합니다. 여러 스킬을 조합하는 심화 주제로 부트캠프에 적합합니다.

| 항목 | 내용 |
| --- | --- |
| Trigger | 행사 일정 확정(D-30), 이후 D-14·D-7·D-1·D+2 단계별 |
| Input | 행사 개요, 참가자 명단(비식별), 안내문 템플릿, 지난 행사 체크리스트·설문 결과 |
| AI 작업 | 단계별 체크리스트 생성, 안내·리마인드 문안 초안, 사후 감사 메일과 설문 요약 리포트 초안 |
| 사람 검토 | 운영 담당자가 각 단계 발송 전 승인 — 단계별 승인 게이트 |
| Output | 단계별 운영 패키지(체크리스트·문안·리포트) |

- **대표 산출물**: 단계별 하위 스킬(안내문·체크리스트·설문 요약)과 이를 묶는 오케스트레이터 워크플로, 단계별 승인 게이트 정의
- **리스크**: 잘못된 일시·링크 발송, 참가자 정보 노출
- **Best Practice 포인트**: 날짜·링크·장소 같은 "치명적 필드"는 발송 전 사람이 대조하는 체크리스트를 별도로 두고, 단계마다 승인자가 다르면 워크플로에 이름으로 명시한다.

## 공통 Best Practice 8가지

모든 주제에 공통 적용되는 원칙입니다. 교육 중 산출물 리뷰와 검증 체크리스트의 기본 항목으로 사용합니다.

1. **사람 검토 없는 자동 발송 금지** — 외부(고객·타 부서)로 나가는 산출물은 반드시 사람 승인 게이트를 거친다. 워크플로 도식에 승인자를 이름 또는 역할로 명시한다.
2. **비식별 샘플로 시작** — 파일럿 전 단계에서는 실데이터 대신 비식별 샘플을 사용하고, 금지 데이터 목록을 스킬에 기록한다.
3. **스킬은 업무 매뉴얼처럼** — 목적, 절차, 금지 사항, 예시 입출력 4요소를 갖추지 않은 스킬은 미완성으로 간주한다. 좋은 예시 1건과 나쁜 예시 1건이 규칙 10줄보다 효과적이다.
4. **`_workspace` 규약 먼저** — 입력·중간·최종 산출물 폴더를 분리하고 명명 규칙을 정한 뒤 실습을 시작한다. 산출물 위치 혼선은 가장 흔한 실패 원인이다.
5. **검증 체크리스트를 산출물보다 먼저 작성** — "무엇이 통과인가"(기대 출력 예시, 필수 포함 항목, 금지 표현)를 먼저 정의하면 스킬 품질이 따라온다.
6. **모르면 멈추고 질문** — 입력이 불충분하거나 판단이 필요한 경우 임의로 채우지 말고 "확인 필요"로 표시하도록 스킬 금지 사항에 명시한다.
7. **근거 인용 강제** — 요약·답변류 산출물은 원문 인용 또는 출처 링크를 필수로 하고, 근거 없는 항목은 삭제하는 규칙을 검증 체크리스트에 넣는다.
8. **버전과 변경 기록** — 스킬은 v1, v1.1처럼 버전을 붙이고 변경 사유를 한 줄씩 기록한다. 규정·양식이 바뀌면 스킬도 함께 바뀌어야 하는 업무 자산임을 팀에 공유한다.
