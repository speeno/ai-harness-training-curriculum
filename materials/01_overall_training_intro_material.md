# 전체 교육 소개 자료

## 하네스(Harness) · Agent Skills 기반 기업 AI 교육

> 공통 소개 세션용 자료  
> 권장 사용 시간: 20~30분  
> 대상: 비개발자 과정, 개발자 과정, 혼합 과정 공통 오프닝

# 1. 자료 목적

이 자료는 하네스(Harness) · Agent Skills 교육의 전체 방향을 설명하기 위한 공통 소개 자료이다.  
비개발자 트랙과 개발자 트랙으로 나뉘기 전에, 모든 수강자가 다음 내용을 같은 언어로 이해하도록 돕는 것을 목표로 한다.

- 왜 AI 활용을 개인 프롬프트 수준에 머물게 두면 안 되는가
- 하네스와 Agent Skills가 조직의 업무 자동화에 어떤 의미를 가지는가
- 비개발자와 개발자가 각각 무엇을 설계하고 산출해야 하는가
- 1일, 3일, 1주 과정은 각각 어떤 상황에 적합한가
- 교육이 끝났을 때 어떤 산출물이 남아야 하는가

## 1.1 전체 자료 패키지 사용 순서

## Slide 1. 하네스(Harness) · Agent Skills 기반 기업 AI 교육

### 화면 구성

**제목**  
하네스(Harness) · Agent Skills 기반 기업 AI 교육

**부제**  
AI 활용을 개인 프롬프트에서 조직의 업무 표준과 개발 하네스로 전환하기

**키워드**

- 업무 자동화
- Agent Skills
- Orchestrator
- `_workspace`
- `SKILL.md`
- 검증과 운영

### 발표자 메모

오늘 교육의 핵심은 단순히 AI 도구를 더 잘 사용하는 방법을 배우는 것이 아닙니다.  
우리가 다루려는 것은 조직 안에서 AI를 반복 가능하게, 안전하게, 그리고 검증 가능하게 쓰는 방법입니다.

개인은 좋은 프롬프트를 만들 수 있습니다. 하지만 조직은 프롬프트만으로 운영되지 않습니다.  
조직에는 절차, 승인, 파일 위치, 책임자, 검증 기준, 보안 정책이 필요합니다.  
이 교육은 그 요소들을 하네스와 Agent Skills라는 구조 안에서 정리하는 과정입니다.

## Slide 2. 오늘의 문제의식: AI 도구는 있지만 운영 체계가 부족하다

### 화면 구성

**핵심 문장**  
많은 조직이 AI 도구를 도입했지만, 실제 업무에서는 성과가 반복되지 않는 경우가 많다.

**현장에서 자주 나타나는 질문**

- 누가 만든 프롬프트가 기준인가?
- 결과물은 어디에 저장해야 하는가?
- 어느 단계에서 사람이 검토해야 하는가?
- 잘못된 결과가 나왔을 때 누가 책임지는가?
- 같은 업무를 다음 달에도 같은 품질로 반복할 수 있는가?

### 발표자 메모

AI 도구 도입 자체는 점점 쉬워지고 있습니다.  
하지만 실제 기업 현장에서는 도구보다 운영 방식이 더 큰 문제가 됩니다.

처음에는 한두 명이 좋은 결과를 냅니다.  
그런데 시간이 지나면 사람마다 프롬프트가 달라지고, 결과물 위치가 달라지고, 승인 절차가 빠지고, 같은 업무를 반복해도 품질이 흔들립니다.

따라서 이 교육은 “AI를 잘 쓰는 개인”보다 “AI를 안정적으로 활용하는 팀”을 만드는 데 초점을 둡니다.

## Slide 3. AI 활용의 흔한 실패 패턴

### 화면 구성

| 실패 패턴 | 현상 | 결과 |
| --- | --- | --- |
| 개인 프롬프트 의존 | 각자 다른 방식으로 요청 | 품질 편차 발생 |
| 산출물 위치 혼선 | 입력·중간·최종 파일이 섞임 | 재현과 감사 어려움 |
| 승인·검토 누락 | AI 결과가 바로 사용됨 | 리스크 증가 |
| 검증 기준 부재 | 맞는지 틀린지 판단 기준 없음 | 파일럿 확산 어려움 |
| 개발 연결 단절 | 현업 요구가 코드로 전달되지 않음 | 자동화 구현 지연 |

### 발표자 메모

AI 자동화가 실패하는 이유는 기술이 부족해서만은 아닙니다.  
대부분의 문제는 업무 기준이 명확하지 않거나, 사람이 개입해야 하는 지점이 정의되지 않았거나, 산출물 관리 규칙이 없어서 발생합니다.

예를 들어 고객 문의 답변 초안을 AI가 작성한다고 해도, 어떤 입력을 기준으로 작성할지, 어떤 표현은 금지할지, 누가 최종 승인할지, 결과물은 어디에 저장할지를 정하지 않으면 실제 업무에 쓰기 어렵습니다.

그래서 우리는 프롬프트보다 더 큰 단위, 즉 Skill, Workflow, Workspace, Verification을 다룹니다.

## Slide 4. 핵심 전환: Prompt에서 Skill로

### 화면 구성

**Before: 개인 프롬프트 중심**

- 개인이 기억하는 요청 방식
- 매번 달라지는 표현
- 파일 위치와 검토 기준 불명확
- 재사용과 인수인계 어려움

**After: 조직 Skill 중심**

- 승인된 업무 절차
- 반복 가능한 입력·출력 기준
- 명확한 금지 사항과 예외 처리
- 검증 가능한 산출물
- 개발 구현으로 연결 가능한 요구사항

### 발표자 메모

프롬프트는 개인의 요청문입니다.  
Skill은 조직이 재사용할 수 있는 업무 절차입니다.

프롬프트가 “이번에 이렇게 해줘”라면, Skill은 “우리 조직에서는 이 업무를 이런 입력, 이런 절차, 이런 검토 기준으로 처리한다”에 가깝습니다.

비개발자에게 Skill은 업무 매뉴얼입니다.  
개발자에게 Skill은 `SKILL.md`와 실행 가능한 자동화 단위입니다.

이 전환이 이루어져야 AI 활용이 개인 역량을 넘어 팀의 자산이 됩니다.

## Slide 5. 하네스(Harness)의 기본 구조

### 화면 구성

```text
사용자 목표
   ↓
에이전트
   ↓
Skill / SKILL.md
   ↓
Orchestrator / Task / Sub-agent
   ↓
_workspace
   ↓
검증 / 사람 승인 / 운영 로그
   ↓
업무 산출물
```

**핵심 설명**

- 사용자는 목표와 맥락을 제공한다.
- 에이전트는 Skill을 참고해 행동한다.
- Orchestrator는 여러 단계를 순서대로 조립한다.
- Task와 Sub-agent는 복잡한 일을 나누어 처리한다.
- `_workspace`는 입력, 중간 결과, 최종 산출물을 구분한다.
- 검증과 승인 절차가 결과물의 신뢰성을 높인다.

### 발표자 메모

하네스는 에이전트가 마음대로 움직이지 않도록 잡아주는 실행 구조라고 이해하면 됩니다.

자동차에 엔진만 있다고 안전하게 달릴 수 없는 것처럼, AI 모델만 있다고 업무 자동화가 완성되지는 않습니다.  
운전대, 브레이크, 계기판, 차선, 정비 규칙이 필요합니다.

기업 AI 활용에서 이 역할을 하는 것이 하네스입니다.  
Skill은 에이전트가 따라야 할 업무 지침이고, Orchestrator는 업무 순서를 조립하며, `_workspace`는 작업 공간을 정리하고, 검증은 결과를 신뢰할 수 있게 만듭니다.

## Slide 6. Agent Skills란 무엇인가

### 화면 구성

**Agent Skill = 에이전트가 특정 업무를 수행하기 위해 참조하는 재사용 가능한 지침 단위**

| 관점 | 의미 |
| --- | --- |
| 비개발자 관점 | 업무 매뉴얼, SOP, 표준 문구, 금지 사항 |
| 개발자 관점 | `SKILL.md`, 트리거 조건, 예시, 실패 시 행동 |
| 조직 관점 | 버전 관리되는 업무 자산 |

**Skill에 포함될 수 있는 내용**

- 업무 목적
- 적용 범위
- 입력 데이터
- 단계별 절차
- 금지 사항
- 예외 처리
- 출력 형식
- 검증 기준

### 발표자 메모

Skill은 단순히 긴 프롬프트가 아닙니다.  
좋은 Skill은 사람이 읽어도 업무 절차로 이해할 수 있고, 에이전트가 읽어도 실행 기준으로 사용할 수 있어야 합니다.

예를 들어 “회의록을 요약해줘”라는 요청은 프롬프트입니다.  
하지만 회의록 요약 Skill은 훨씬 구체적입니다.

어떤 회의록을 대상으로 하는지, 어떤 항목을 반드시 추출해야 하는지, 결정사항과 액션 아이템을 어떻게 구분할지, 민감한 내용은 어떻게 처리할지, 최종 결과는 어디에 저장할지를 포함해야 합니다.

## Slide 7. 비개발자 트랙: 업무를 자동화 가능한 요구사항으로 정리한다

### 화면 구성

**핵심 질문**  
우리 팀 업무를 어떤 순서와 규칙으로 자동화할 것인가?

**비개발자 트랙의 역할**

- 자동화 후보 업무 선정
- 업무 시나리오 정의
- Skill을 업무 매뉴얼로 작성
- 승인·검토 워크플로 설계
- `_workspace` 폴더 규칙 정의
- 리스크와 검증 체크리스트 작성

**대표 산출물**

- 업무 시나리오 1페이지
- Skill 초안
- 워크플로 다이어그램
- `_workspace` 운영 규칙
- 리스크 레지스터
- 검증 체크리스트

### 발표자 메모

비개발자 트랙의 핵심은 코드를 작성하는 것이 아닙니다.  
현업이 알고 있는 업무 지식을 자동화 가능한 형태로 정리하는 것입니다.

현업은 어떤 업무가 반복되는지, 어디에서 실수가 나는지, 누가 승인해야 하는지, 어떤 표현이나 데이터가 위험한지를 가장 잘 알고 있습니다.

따라서 비개발자 트랙에서는 “AI에게 뭘 시킬까?”보다 “우리 업무를 어떤 규칙으로 설명해야 반복 가능할까?”를 다룹니다.

## Slide 8. 개발자 트랙: 요구사항을 실행 가능한 하네스로 구현한다

### 화면 구성

**핵심 질문**  
에이전트가 안전하게 도구를 쓰고, Skill을 재사용·검증하려면 무엇을 정의해야 하는가?

**개발자 트랙의 역할**

- 에이전트 역할과 도구 범위 정의
- `SKILL.md` 구조화
- 오케스트레이터 스킬 작성
- Task와 Sub-agent 분해
- `AGENTS.md` 포인터 작성
- 검증 스크립트와 로그 기준 설계
- 보안, 시크릿, 롤백 기준 반영

**대표 산출물**

- `SKILL.md`
- 오케스트레이터 스킬
- Task 설계 문서
- `AGENTS.md`
- 스모크 검증 절차
- 롤아웃 체크리스트

### 발표자 메모

개발자 트랙은 현업 요구사항을 실제 실행 구조로 연결합니다.

좋은 자동화는 에이전트가 한 번 답을 잘하는 것에서 끝나지 않습니다.  
반복 실행해도 안정적이어야 하고, 실패했을 때 원인을 추적할 수 있어야 하며, 보안 정책을 지켜야 하고, 팀원이 유지보수할 수 있어야 합니다.

따라서 개발자 트랙에서는 Skill을 코드와 문서가 만나는 지점으로 다룹니다.  
`SKILL.md`, 오케스트레이터, Task, Sub-agent, `AGENTS.md`, 검증 스크립트가 그 핵심 요소입니다.

## Slide 9. 비개발자와 개발자의 연결 구조

### 화면 구성

```text
비개발자 산출물
업무 시나리오 / Skill 초안 / 워크플로 / 리스크 / 검증 기준
        ↓
개발자 입력
fixture / acceptance criteria / SKILL.md / Task 설계 / 테스트 기준
        ↓
실행 가능한 하네스
에이전트 / 오케스트레이터 / 검증 / 로그 / 운영
```

**핵심 메시지**

비개발자 트랙은 요구사항을 만든다.  
개발자 트랙은 그 요구사항을 실행 가능하고 검증 가능한 구조로 만든다.

### 발표자 메모

이 교육의 중요한 특징은 비개발자 과정과 개발자 과정이 따로 노는 것이 아니라 서로 연결된다는 점입니다.

비개발자가 만든 업무 시나리오는 개발자에게 단순 설명 자료가 아닙니다.  
그 자체가 테스트 입력, 인수 조건, Skill 구조, 검증 체크리스트의 기반이 됩니다.

반대로 개발자는 현업 요구사항을 코드로만 해석하지 않고, 어떤 단계에 사람 승인이 필요한지, 어떤 산출물이 보존되어야 하는지, 어떤 실패 모드가 중요한지를 함께 반영해야 합니다.

1주 과정은 회사 차원의 표준화와 운영 체계를 준비하는 과정입니다.  
이때는 단일 업무 자동화가 아니라, 분기별 개정, 보안, 운영 로그, 롤백, 로드맵까지 포함합니다.

# 4. 공통 소개용 한 장 요약

## 하네스 · Agent Skills 교육의 핵심

이 교육은 AI 도구 사용법 교육이 아니라, 조직이 AI 자동화를 안전하고 반복 가능하게 운영하기 위한 업무 표준화 및 개발 하네스 구축 교육이다.

| 구분 | 핵심 내용 |
| --- | --- |
| 문제 | 개인별 프롬프트, 산출물 혼선, 승인 누락, 검증 부재 |
| 전환 | Prompt 중심에서 Skill 중심으로 전환 |
| 비개발자 역할 | 업무 시나리오, Skill 초안, 워크플로, `_workspace` 규칙 작성 |
| 개발자 역할 | `SKILL.md`, 오케스트레이터, Task, `AGENTS.md`, 검증 구현 |
| 1일 과정 | 공통 언어 정렬과 최소 초안 작성 |
| 3일 과정 | 파일럿 수준의 설계 패키지 완성 |
| 1주 과정 | 조직 표준, 거버넌스, 운영 체계 수립 |
| 최종 목표 | AI 활용을 개인 역량이 아니라 조직 자산으로 전환 |

# 5. 강사용 오프닝 멘트 예시

안녕하세요. 오늘 교육은 AI 도구의 기능을 하나씩 배우는 시간이 아닙니다.  
이미 많은 조직이 다양한 AI 도구를 사용하고 있지만, 실제 현장에서는 여전히 비슷한 문제가 반복됩니다.

누군가는 좋은 프롬프트로 좋은 결과를 만들지만, 그 방식이 팀에 공유되지 않습니다.  
결과물이 어디에 저장됐는지 알기 어렵고, 승인이나 검토 단계가 빠지기도 합니다.  
또 같은 업무를 다음 달에 다시 실행했을 때 같은 품질이 나올지 확신하기 어렵습니다.

그래서 오늘 우리가 다룰 핵심은 프롬프트가 아니라 Skill입니다.  
Skill은 AI에게 일을 시키기 위한 단순 요청문이 아니라, 조직이 합의한 업무 절차, 금지 사항, 입력과 출력 기준, 검증 방식을 담은 재사용 가능한 업무 자산입니다.

비개발자 과정에서는 이 Skill을 업무 매뉴얼과 워크플로로 정리합니다.  
개발자 과정에서는 이를 `SKILL.md`, 오케스트레이터, Task, `AGENTS.md`, 검증 파이프라인으로 연결합니다.

오늘의 목표는 완벽한 자동화 시스템을 하루 만에 만드는 것이 아닙니다.  
대신 우리 조직이 AI 자동화를 어떤 기준으로 설계하고, 어떤 산출물을 남기며, 어떻게 검증할 것인지에 대한 공통 언어를 만드는 것입니다.

              ↓
