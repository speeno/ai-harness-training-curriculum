# 07. 평가 체크리스트

> 과정명: 하네스(Harness) · Agent Skills 기반 기업 AI 교육  
> 적용 범위: 공통 소개, 비개발자 1일 과정, 개발자 1일 과정, 후속 3일·1주 과정 확장 평가  
> 목적: 교육 후 산출물이 실제 파일럿, 개발자 트랙, 조직 표준화 논의로 이어질 수 있는지 평가한다.

---

# 1. 평가 체크리스트의 목적

이 체크리스트는 수강생을 점수로 줄 세우기 위한 자료가 아닙니다.  
교육 후 만들어진 산출물이 다음 단계로 이동할 준비가 되었는지 확인하기 위한 품질 점검 자료입니다.

평가의 핵심 질문은 다음과 같습니다.

```text
이 산출물은 다른 사람이 읽어도 같은 업무와 같은 실행 흐름을 이해할 수 있는가?
```

```text
이 산출물은 실제 파일럿 또는 개발 구현으로 넘길 수 있을 만큼 입력, 절차, 출력, 검토 기준이 명확한가?
```

```text
이 산출물은 AI 자동화의 리스크와 검증 기준을 충분히 포함하고 있는가?
```

---

# 2. 평가 등급 체계

각 항목은 아래 4단계로 평가합니다.

| 등급 | 의미 | 설명 |
| --- | --- | --- |
| 0 | 미작성 | 해당 항목이 없거나 확인할 수 없음 |
| 1 | 초안 수준 | 항목은 있으나 모호하거나 누락이 많음 |
| 2 | 활용 가능 | 기본 요건을 충족하며 팀 내부 검토 가능 |
| 3 | 파일럿 가능 | 실제 파일럿 또는 개발자 트랙으로 넘길 수 있는 수준 |

권장 판정 기준:

| 평균 점수 | 판정 | 의미 |
| --- | --- | --- |
| 0.0~0.9 | 보완 필요 | 핵심 산출물이 부족함 |
| 1.0~1.9 | 초안 완성 | 팀 내부 논의는 가능하나 추가 보완 필요 |
| 2.0~2.5 | 실무 활용 가능 | 파일럿 검토 대상으로 적합 |
| 2.6~3.0 | 파일럿 준비 우수 | 개발자 트랙 또는 파일럿 브랜치로 전환 가능 |

## 2.1 평가 후 보완 액션 매트릭스

평가 결과는 점수로 끝내지 않고, 다음 보완 작업으로 연결해야 한다. 아래 표는 낮은 점수가 나온 영역을 어떤 자료와 활동으로 보완할지 정리한 기준이다.

| 낮은 점수가 나온 영역 | 대표 원인 | 보완할 자료 | 권장 보완 액션 |
| --- | --- | --- | --- |
| 업무 후보·시나리오 | 범위가 크거나 Trigger/Input/Output이 모호함 | `03_non_developer_1_day_workbook_complete.md` | 업무 후보를 1건으로 좁히고 시나리오 1페이지를 다시 작성 |
| Skill 초안 | 절차, 금지 사항, 출력 형식이 부족함 | `02_non_developer_1_day_slides_recreated.md`, `03_non_developer_1_day_workbook_complete.md` | Purpose, Input, Procedure, Constraint, Review 기준을 보강 |
| Workflow | 승인자, 예외 흐름, 재작업 경로가 없음 | `06_instructor_operation_guide.md`, `08_three_day_course_expansion_material.md` | 정상 흐름과 예외 흐름을 분리하고 human gate 추가 |
| Workspace | 원본·초안·최종본 위치가 섞임 | `03_non_developer_1_day_workbook_complete.md` | in/work/out/archive/tmp 용도와 파일명 규칙 작성 |
| Risk·Verification | 검증 항목이 일반적이고 통과 기준이 없음 | `07_evaluation_checklist.md`, `08_three_day_course_expansion_material.md` | 샘플 입력, 기대 출력, 통과/실패 예시 추가 |
| 개발자 산출물 | `SKILL.md`, Orchestrator, Task가 서로 연결되지 않음 | `04_developer_1_day_slide_material.md`, `05_developer_1_day_practice_templates.md` | 비개발자 산출물을 acceptance criteria와 smoke test로 전환 |
| 보안·운영 | PII, 로그, 승인 기록, 롤백 기준이 없음 | `05_developer_1_day_practice_templates.md`, `09_one_week_course_expansion_material.md` | Security Notes, Logging Standard, Runbook 초안 작성 |

보완 우선순위는 다음 순서로 잡는다.

```text
1. 업무 범위와 입력·출력 명확화
2. 사람 검토와 승인 지점 확정
3. 검증 기준과 보안 기준 보강
4. 개발자 산출물과 운영 문서 연결
```
