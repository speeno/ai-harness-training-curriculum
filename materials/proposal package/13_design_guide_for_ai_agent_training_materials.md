# 13. AI 에이전트 교육 자료 디자인 가이드

> 적용 대상: PPT, 제안서, One-page Summary, 브로셔, 인포그래픽, 워크북 표지, 강사용 자료  
> 주제: 하네스(Harness) · Agent Skills 기반 기업 AI 교육  
> 목표: 일관적이고 신뢰감 있는 B2B 교육·컨설팅 자료 디자인 체계 수립

---

# 1. 디자인 가이드의 목적

이 디자인 가이드는 이후 제작할 모든 PPT와 디자인 자료의 시각적 일관성을 확보하기 위한 기준 문서입니다.

자료의 핵심 메시지는 다음입니다.

```text
AI 활용을 개인별 프롬프트 중심에서
조직이 재사용할 수 있는 Skill, Workflow, Workspace, Verification 체계로 전환한다.
```

따라서 디자인은 다음 인상을 주어야 합니다.

- 신뢰감
- 체계성
- 실무성
- 안정성
- 확장 가능성
- 기술적 전문성
- 교육 자료로서의 명확성

이 자료는 화려한 AI 홍보 이미지보다, **조직 운영·표준화·검증·하네스 구조**를 시각적으로 명확히 전달하는 데 초점을 둡니다.

---

# 2. 전체 디자인 원칙

## 2.1 핵심 디자인 원칙

| 원칙 | 설명 |
| --- | --- |
| 신뢰 우선 | 과장된 AI 이미지보다 정돈된 구조와 명확한 정보 전달을 우선한다. |
| 구조 중심 | 개념 간 관계, 흐름, 단계, 산출물 구조를 시각화한다. |
| 실무형 톤 | 추상적인 미래 비전보다 실제 업무 적용과 산출물 중심으로 표현한다. |
| 일관된 용어 | Skill, Workflow, Workspace, Verification, Orchestrator 등의 표현을 통일한다. |
| 한 장 한 메시지 | 슬라이드 한 장에는 하나의 핵심 메시지만 담는다. |
| 과도한 장식 배제 | 복잡한 배경, AI 로봇 이미지, 과한 네온 효과를 피한다. |
| 검증 가능성 강조 | 체크리스트, 게이트, 로그, 버전, 승인 흐름을 시각 요소로 반복 사용한다. |

---

## 2.2 디자인 키워드

| 키워드 | 의미 |
| --- | --- |
| Structured | 업무와 기술 요소가 체계적으로 정리된 느낌 |
| Practical | 실제 교육과 실습에 바로 사용할 수 있는 느낌 |
| Governed | 승인, 검토, 보안, 검증이 포함된 안정적 운영 느낌 |
| Reusable | Skill과 템플릿이 조직 자산으로 반복 사용되는 느낌 |
| Technical but Accessible | 개발자에게는 충분히 기술적이고, 비개발자에게도 이해 가능한 느낌 |

# 3. 브랜드 톤 & 무드

## 3.1 권장 톤

| 항목 | 방향 |
| --- | --- |
| 전체 분위기 | B2B 컨설팅 + 기술 교육 + 내부 표준화 문서의 중간 톤 |
| 감정 | 안정감, 전문성, 신뢰, 실행 가능성 |
| 표현 방식 | 과장보다 정리, 추상보다 구조, 홍보보다 설계 |
| 시각 언어 | 파이프라인, 카드, 체크리스트, 로드맵, 폴더 구조, 게이트 |

---

## 3.2 피해야 할 톤

| 피해야 할 표현 | 이유 |
| --- | --- |
| 로봇, 인간형 AI 캐릭터 중심 | 기업 표준·거버넌스 교육의 신뢰감을 약화할 수 있음 |
| 과도한 네온·사이버펑크 스타일 | 실무 교육보다 이벤트성 홍보처럼 보일 수 있음 |
| 너무 많은 그라데이션 | 정보 구조가 흐려질 수 있음 |
| 아이콘 남발 | 핵심 개념보다 장식이 강조될 수 있음 |
| 지나치게 개발자 전용 UI | 비개발자 과정에서 접근성이 낮아짐 |
| 추상적 AI 배경 이미지 | 실제 산출물 중심이라는 교육 취지와 맞지 않음 |

## 4.3 색상 비율

권장 비율:

```text
Neutral / White 계열: 70%
Navy / Slate 계열: 20%
Accent 색상: 10%
```

사용 원칙:

- Accent 색상은 강조할 때만 사용한다.
- 한 슬라이드에서 Accent 색상은 1~2개만 사용한다.
- Green, Amber, Red는 의미가 분명할 때만 사용한다.
- 표 전체를 색칠하지 않고, 헤더·상태·아이콘에 제한적으로 사용한다.

# 5. 타이포그래피

## 5.1 권장 폰트

실제 제작 환경에 따라 아래 중 하나를 사용한다.

| 언어 | 권장 폰트 | 대체 폰트 |
| --- | --- | --- |
| 한글 | Pretendard | Noto Sans KR, 맑은 고딕 |
| 영문 | Inter | Aptos, Segoe UI, Arial |
| 코드 | JetBrains Mono | Consolas, Courier New |

문서 공유 호환성을 고려하면 PowerPoint에서는 다음 조합이 안정적이다.

```text
한글/영문 본문: Aptos 또는 맑은 고딕
코드/파일 경로: Consolas
```

브랜드형 자료나 PDF 변환 자료는 다음 조합을 권장한다.

```text
한글/영문: Pretendard 또는 Noto Sans KR
코드: JetBrains Mono
```

| 규칙 | 설명 |
| --- | --- |
| 제목은 짧게 | 한 줄 또는 최대 두 줄 |
| 본문은 문장보다 키워드 | 긴 문단보다 카드·표·단계로 분리 |

| 강조는 굵게 또는 색상 하나만 | 굵게 + 색상 + 밑줄을 동시에 쓰지 않음 |

# 6. 레이아웃 시스템

## 6.1 기본 슬라이드 규격

| 항목 | 기준 |
| --- | --- |
| 화면비 | 16:9 Wide |
| 기본 배경 | Soft White 또는 White |
| 좌우 여백 | 최소 0.6 inch 이상 |
| 상단 제목 영역 | 전체 높이의 15~20% |
| 본문 영역 | 카드, 표, 다이어그램 중심 |
| 하단 영역 | 페이지 번호, 섹션명, 짧은 보조 정보 |

---

## 6.2 권장 레이아웃 유형

### Layout A. 제목 + 3카드

사용 상황:

- 세 가지 개념 설명
- 트랙 비교
- 기대 효과 요약

구조:

```text
[Title]

[Card 1]   [Card 2]   [Card 3]
```

예시:

- Skill / Workflow / Workspace
- 1일 / 3일 / 1주
- 비개발자 / 개발자 / 혼합

---

### Layout B. Before / After 비교

사용 상황:

- Prompt 중심 → Skill 중심
- 개인 사용 → 조직 표준
- 초안 → 파일럿 패키지

구조:

```text
[Title]

Before                     After
[Problem Cards]     →      [Improved Cards]
```

---

### Layout C. 파이프라인 흐름

사용 상황:

- Harness 구조
- 비개발자 산출물 → 개발자 산출물
- 교육 → 파일럿 → 확산

구조:

```text
[Step 1] → [Step 2] → [Step 3] → [Step 4] → [Step 5]
```

---

### Layout D. 일정표 / 로드맵

사용 상황:

- 1일 과정 시간표
- 3일 과정 Day별 흐름
- 1주 과정 5일 흐름
- 30/60/90일 로드맵

구조:

```text
Day 1 | Day 2 | Day 3 | Day 4 | Day 5
```

---

### Layout E. 표 + 핵심 코멘트

사용 상황:

- 과정 비교
- 산출물 비교
- 평가 기준
- 준비사항

구조:

```text
[Title]
[Short message]
[Table]
```

---

### Layout F. 폴더 트리 / 코드 블록

사용 상황:

- `_workspace` 구조
- 개발자 레포 구조
- `AGENTS.md` 구조
- `SKILL.md` 템플릿

구조:

```text
[Title]
[Left explanation]   [Right code/folder block]
```

# 8. 다이어그램 디자인 규칙

## 8.1 Harness 구조도

기본 구조:

```text
Business Scenario
   ↓
Agent
   ↓
Skill / SKILL.md
   ↓
Orchestrator / Task / Sub-agent
   ↓
Workspace
   ↓
Verification / Human Review
   ↓
Approved Output
```

디자인 규칙:

- 세로형 또는 가로형 파이프라인 사용
- 각 박스에 아이콘 1개 + 키워드 1개 + 짧은 설명 1개
- Verification과 Human Review는 Green 또는 Amber로 강조
- Approved Output은 최종 도착점으로 별도 강조

---

## 8.2 Prompt → Skill 전환도

Before:

- 개인 프롬프트
- 일회성 요청
- 결과 품질 편차
- 승인·검증 없음

After:

- 조직 Skill
- 재사용 가능한 절차
- 입력·출력 기준
- 승인·검증 포함

디자인:

```text
[Prompt 중심]  →  [Skill 중심]
```

Before는 Gray 계열, After는 Navy + Blue 계열 사용.

---

## 8.3 비개발자 → 개발자 연결도

```text
업무 시나리오       →  Acceptance Criteria
Skill 초안          →  SKILL.md
Workflow            →  Orchestrator
리스크              →  Failure Matrix
검증 체크리스트      →  Smoke Verification
```

디자인:

- 좌측 비개발자 산출물은 Light Blue 카드
- 우측 개발자 산출물은 Slate/Navy 카드
- 가운데 화살표는 Trust Blue
- 상단 제목: “현업 요구사항이 개발 하네스로 전환되는 구조”

---

## 8.4 과정 선택 가이드

구조:

```text
AI 도입 초기? → 1일 과정
파일럿 준비? → 3일 과정
조직 표준화? → 1주 과정
```

디자인:

- 의사결정 플로우차트 또는 3단 카드
- 1일은 Light Blue, 3일은 Navy, 1주는 Green/Navy 조합
- 각 카드에 “추천 상황 / 산출물 / 다음 단계” 포함

---

## 8.5 Verification Pyramid

구조:

```text
Level 4: Optional E2E
Level 3: Regression Fixture
Level 2: Smoke Check
Level 1: Static / Structure Check
```

디자인:

- 피라미드 또는 계단형
- 아래로 갈수록 넓고 기본적인 검증
- 위로 갈수록 고도화된 검증
- Green 계열은 통과, Amber는 검토 필요로 사용

# 9. PPT별 디자인 적용 지침

## 9.1 제안 발표 PPT

| 항목 | 지침 |
| --- | --- |
| 톤 | 가장 고급스럽고 간결하게 |
| 텍스트 | 슬라이드당 3~5개 핵심 bullet 이하 |
| 시각화 | 문제→해결, 과정 선택, 산출물 맵 중심 |
| 색상 | Navy 중심, Accent Blue 제한 사용 |
| 표 | 단순화하고 핵심 비교만 사용 |
| 마무리 | 기대효과와 다음 단계 강조 |

---

## 9.2 공통 교육 소개 PPT

| 항목 | 지침 |
| --- | --- |
| 톤 | 교육용이지만 전문적으로 |
| 텍스트 | 강사가 설명할 수 있도록 핵심 키워드 중심 |
| 시각화 | Harness 구조와 트랙 연결도 중심 |
| 실습 요소 | 최소화하고 개념 정렬에 집중 |
| 반복 메시지 | Prompt → Skill, 개인 → 조직 표준 |

---

## 9.3 비개발자 1일 PPT

| 항목 | 지침 |
| --- | --- |
| 톤 | 친절하고 실무적으로 |
| 텍스트 | 기술 용어를 쉬운 표현과 함께 사용 |

# 16. 최종 권장 방향

이 교육 자료의 디자인은 “화려한 AI 세미나”보다 “신뢰할 수 있는 기업 AI 운영 체계 교육”에 가까워야 한다.

따라서 다음 방향을 유지한다.

```text
Less hype, more structure.
Less decoration, more clarity.
Less prompt trick, more reusable skill.
Less demo, more governance-ready output.
```

한국어 표현으로는 다음과 같이 정리할 수 있다.

```text
과장보다 구조,
장식보다 명확성,
프롬프트 요령보다 재사용 가능한 Skill,
일회성 데모보다 운영 가능한 산출물.
```
