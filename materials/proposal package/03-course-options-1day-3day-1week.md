# 과정 선택 가이드: 1일 · 3일 · 1주

## 1. 과정 선택 기준

과정은 조직의 AI 자동화 준비 수준과 목표에 따라 선택합니다.

| 질문 | 추천 과정 |
| --- | --- |
| AI 자동화의 공통 언어를 먼저 맞추고 싶은가? | 1일 과정 |
| 특정 팀의 파일럿 업무를 구체화해야 하는가? | 3일 과정 |
| 조직 표준, 거버넌스, 운영 체계까지 만들고 싶은가? | 1주 과정 |
| 현업과 개발자가 함께 파일럿을 준비해야 하는가? | 3일 혼합 과정 |
| 플랫폼 조직이 에이전트 하네스 표준을 만들고 싶은가? | 개발자 1주 과정 |

---

## 2. 1일 과정: 공통 언어와 최소 초안

### 추천 상황

- AI 자동화 도입 초기
- 조직 내 공통 언어가 필요한 경우
- 특정 업무 1건을 빠르게 파일럿 후보로 만들고 싶은 경우
- 3일 또는 1주 과정 전에 참가자 이해 수준을 맞추고 싶은 경우

### 교육 목표

- Skill, Workflow, Workspace, Verification의 의미 이해
- 반복 업무 1건 선정
- 업무 시나리오와 Skill 초안 작성
- 개발자 과정에서는 `SKILL.md`, Orchestrator, Task, `AGENTS.md`, Smoke Verification 초안 작성

### 대표 결과물

| 트랙 | 결과물 |
| --- | --- |
| 비개발자 | 업무 시나리오 1페이지, Skill 초안, Workflow, Workspace 규칙 |
| 개발자 | `SKILL.md`, Orchestrator 초안, Task 노트, `AGENTS.md`, Smoke Verification 메모 |

---

## 3. 3일 과정: 파일럿 설계 패키지

### 추천 상황

- 특정 부서 또는 팀 단위 AI 자동화 파일럿 준비
- 업무 후보를 여러 개 비교해야 하는 경우
- 승인자, 검토자, 산출물 위치, 검증 기준을 명확히 해야 하는 경우
- 개발자와 현업의 요구사항 연결이 필요한 경우

### 교육 목표

- 자동화 후보 3~5건 우선순위화
- 파일럿 대상 1건 선정
- Skill v1 또는 Skill Package 작성
- 승인·예외 Workflow와 Orchestrator 설계
- 리스크와 Failure Matrix 작성
- Smoke Verification과 Rollout Checklist 작성

### 대표 결과물

- 자동화 후보 우선순위 표
- 파일럿 설계서
- Skill v1 / Skill Package
- Workflow / Orchestrator
- Risk Register / Failure Matrix
- Validation Checklist / Smoke Verification
- Rollout Checklist

---

## 4. 1주 과정: 조직 표준과 운영 모델

### 추천 상황

- 전사 또는 부서 단위 AI 자동화 표준 수립
- 보안, 검증, 승인, 감사 가능성이 중요한 조직
- 파일럿 이후 확산 로드맵이 필요한 경우
- 플랫폼·DevEx 조직이 에이전트 하네스 표준을 만들고 싶은 경우

### 교육 목표

- 자동화 포트폴리오 작성
- Skill v1.x와 표준 템플릿 작성
- Workflow, RACI, SLA, 에스컬레이션 기준 수립
- Workspace 운영 가이드와 감사 가능성 확보
- 개발자 참조 구현, 검증 파이프라인, Runbook, SLO 초안 작성
- 30/60/90일 로드맵 수립

### 대표 결과물

- 자동화 포트폴리오
- 조직 Skill 표준
- 참조 구현 레포 또는 브랜치
- 검증 파이프라인 초안
- Runbook
- SLO Draft
- 스폰서 브리프
- 30/60/90일 로드맵

## 5. 조직 상황별 추천 조합

| 조직 상황 | 추천 조합 | 이유 |
| --- | --- | --- |
| AI 도입 초기 | 비개발자 1일 + 개발자 1일 | 공통 언어와 기본 산출물 확보 |
| 특정 부서 파일럿 준비 | 비개발자 3일 + 개발자 3일 | 업무 요구사항과 기술 설계를 함께 구체화 |
| 현업 요구는 있으나 구현 표준이 없음 | 비개발자 3일 후 개발자 3일 | 요구사항 패키지를 기술 산출물로 전환 |
| 플랫폼 조직이 먼저 표준을 만들고 싶음 | 개발자 1주 후 비개발자 1일/3일 확산 | 기술 표준을 먼저 만든 뒤 현업 확산 |
| 전사 표준·거버넌스 준비 | 비개발자 1주 + 개발자 1주 | 운영 모델, 보안, 검증, 로드맵까지 포함 |
| 보안·컴플라이언스 이슈가 큰 조직 | 1주 과정 권장 | 검증, 보존, 승인, 감사 가능성 포함 필요 |

## 6. 과정별 산출물 비교

| 산출물 | 1일 | 3일 | 1주 |
| --- | --- | --- | --- |
| 업무 후보 | 1건 | 3~5건 비교 후 1건 선정 | 5건 이상 포트폴리오 |
| Skill | 초안 | Skill v1 | Skill v1.x와 표준 템플릿 |
| Workflow | 기본 흐름 | 승인·예외·재작업 포함 | SLA, RACI, 에스컬레이션 포함 |
| Workspace | 기본 폴더 규칙 | 버전·보존 규칙 포함 | 접근 권한, 감사 가능성 포함 |
| 검증 | 체크리스트 | 샘플 입력·기대 출력 | 회귀 샘플과 운영 점검 포함 |
| 개발 산출물 | 기본 템플릿 | Skill Package, Smoke Test | 참조 구현, CI/검증, Runbook |
| 운영 산출물 | 다음 액션 | Rollout Checklist | 30/60/90일 로드맵 |

## 7. 단계적 도입 로드맵

1. **1일 도입 워크숍**  
   공통 언어와 최소 산출물을 만듭니다.

2. **3일 파일럿 설계 과정**  
   특정 업무를 파일럿 설계 패키지로 구체화합니다.

3. **파일럿 실행**  
   샘플 데이터와 제한된 업무 범위에서 검증합니다.

4. **1주 조직 표준 과정**  
   Skill 표준, 검증 체계, 운영 런북, 확장 로드맵을 만듭니다.

5. **전사 확산**  
   검증된 Skill과 Workflow를 다른 부서로 확장합니다.
