# 06. 강사용 운영 가이드

> 과정명: 하네스(Harness) · Agent Skills 기반 기업 AI 교육  
> 적용 범위: 공통 소개, 비개발자 1일 과정, 개발자 1일 과정  
> 목적: 강사가 교육을 안정적으로 진행하고, 수강생이 실제 산출물을 완성하도록 돕기 위한 운영 가이드

---

# 1. 이 가이드의 목적

이 문서는 강사가 교육을 진행할 때 참고하는 운영 자료입니다.  
슬라이드나 워크북과 달리, 실제 교육장에서 다음 사항을 관리하는 데 초점을 둡니다.

- 교육 시작 전 준비사항
- 시간대별 운영 방식
- 강의와 실습의 균형
- 수강생 질문 대응
- 실습 중간 점검
- 발표 및 피드백 운영
- 교육 종료 후 후속 액션 안내

이 교육은 단순한 AI 도구 사용법 강의가 아니라, 조직이 AI 자동화를 반복 가능하게 운영하기 위한 **업무 표준화와 개발 하네스 구축 교육**입니다.  
따라서 강사는 기능 설명보다 **산출물 완성**과 **팀 합의 가능성**에 초점을 맞춰야 합니다.

# 2. 전체 운영 원칙

## 2.1 핵심 운영 원칙

| 원칙 | 설명 |
| --- | --- |
| 산출물 중심 | 수강생이 교육 후 바로 활용할 수 있는 문서나 템플릿을 남기게 한다. |
| 작은 업무 선택 | 너무 큰 자동화 주제를 고르지 않게 한다. 하루 과정에서는 1개 업무만 다룬다. |
| 공통 언어 반복 | Skill, Workflow, Workspace, Verification의 의미를 반복해서 정렬한다. |
| 사람 검토 강조 | AI가 모든 것을 자동 처리하는 것이 아니라, 사람 승인과 검토가 포함되어야 함을 강조한다. |
| 보안·검증 기본 포함 | 민감정보, 원본 보호, 검증 기준은 최소 수준이라도 반드시 작성하게 한다. |
| 완성도보다 명확성 | 문장이 세련되지 않아도 입력·출력·절차·책임이 명확하면 좋은 초안으로 본다. |

## 2.2 운영 중 위험 신호와 개입 기준

강사는 수강생 산출물이 아래 신호를 보이면 즉시 범위를 좁히거나 질문을 통해 구조를 다시 잡아야 한다.

| 위험 신호 | 원인 | 강사 개입 방식 | 목표 상태 |
| --- | --- | --- | --- |
| 업무 범위가 부서 전체로 커짐 | 자동화 대상을 시스템 전체로 이해함 | “오늘 하루 안에 입력·출력·검토자를 설명할 수 있는 한 업무로 줄이면 무엇인가?”라고 질문한다. | 업무 1건으로 축소 |
| Skill이 프롬프트 문장만 있음 | 목적·입력·출력 계약이 빠짐 | Purpose, Input, Procedure, Constraint, Review 항목으로 다시 나누게 한다. | 업무 매뉴얼 형태 |
| Workflow에 승인자가 없음 | AI 결과를 바로 사용한다고 가정함 | “최종 책임자는 누구이며, 반려되면 어디로 돌아가는가?”를 묻는다. | human gate 포함 |
| Workspace 규칙이 비어 있음 | 파일 관리가 품질과 무관하다고 생각함 | 원본, 초안, 최종본이 섞였을 때 재현이 불가능한 사례를 설명한다. | in/work/out 구분 |
| 검증 항목이 “좋아 보임” 수준 | 통과 기준이 없음 | 누락, 왜곡, 개인정보, 승인, 저장 위치를 최소 검증 축으로 제시한다. | 체크 가능한 기준 |
| 개발자 Task가 과도하게 쪼개짐 | 역할 경계보다 기술 분해에 집중함 | I/O와 실패 복구 가치가 있는 단위만 Task로 남기게 한다. | 3~5개 Task |
| 모든 실패를 자동 재시도함 | 실패 유형을 구분하지 않음 | Retry, Stop, Human Review, Escalation을 나누어 쓰게 한다. | 실패별 대응 분리 |

## 2.3 강사가 반복해서 말해야 할 핵심 문장

교육 중 아래 문장을 반복해서 사용하면 수강생의 이해를 안정시킬 수 있습니다.

```text
오늘의 목표는 완벽한 자동화가 아니라, 합의 가능한 자동화 초안을 만드는 것입니다.
```

```text
Skill은 긴 프롬프트가 아니라, 팀이 재사용할 수 있는 업무 매뉴얼입니다.
```

```text
AI가 할 일과 사람이 검토할 일을 구분해야 실제 업무에 사용할 수 있습니다.
```

```text
입력, 중간 결과, 최종 산출물이 섞이면 나중에 재현과 검증이 어렵습니다.
```

```text
개발자 트랙의 목표는 AI가 한 번 잘 답하게 하는 것이 아니라, 반복 실행 가능한 하네스를 만드는 것입니다.
```

# 4. 공통 소개 세션 운영 가이드

## 4.1 권장 시간

20~30분

## 4.2 목표

공통 소개 세션의 목표는 비개발자와 개발자가 같은 언어로 교육을 시작하도록 만드는 것입니다.

수강생이 이해해야 할 내용:

- AI 도구 도입만으로는 반복 가능한 성과가 나오기 어렵다.
- 개인 프롬프트 중심 활용은 조직 표준으로 확장되기 어렵다.
- Skill은 조직이 재사용할 수 있는 업무 자산이다.
- 비개발자와 개발자는 서로 다른 산출물을 만들지만 연결되어야 한다.
- 교육의 최종 목표는 산출물을 남기는 것이다.

---

## 4.3 공통 소개 진행 흐름

| 순서 | 내용 | 강사 포인트 |
| --- | --- | --- |
| 1 | 교육 목적 소개 | AI 도구 사용법이 아니라 운영 체계 교육임을 강조 |
| 2 | 실패 패턴 제시 | 개인 프롬프트, 파일 혼선, 승인 누락, 검증 부재 |
| 3 | 핵심 전환 설명 | Prompt에서 Skill로 전환 |
| 4 | 하네스 구조 설명 | Agent, Skill, Workflow, Workspace, Verification 연결 |
| 5 | 트랙 차이 설명 | 비개발자 = 요구사항, 개발자 = 실행 구조 |
| 6 | 오늘 산출물 안내 | 교육 종료 시 무엇이 남아야 하는지 확인 |

---

## 4.4 공통 소개 오프닝 멘트

```text
오늘 교육은 AI 도구의 기능을 하나씩 배우는 시간이 아닙니다.
이미 많은 조직이 다양한 AI 도구를 사용하고 있지만, 실제 현장에서는 여전히 비슷한 문제가 반복됩니다.

누군가는 좋은 프롬프트로 좋은 결과를 만들지만, 그 방식이 팀에 공유되지 않습니다.
결과물이 어디에 저장됐는지 알기 어렵고, 승인이나 검토 단계가 빠지기도 합니다.
또 같은 업무를 다음 달에 다시 실행했을 때 같은 품질이 나올지 확신하기 어렵습니다.

그래서 오늘 우리가 다룰 핵심은 프롬프트가 아니라 Skill입니다.
Skill은 AI에게 일을 시키기 위한 단순 요청문이 아니라,
조직이 합의한 업무 절차, 금지 사항, 입력과 출력 기준, 검증 방식을 담은 재사용 가능한 업무 자산입니다.
```

# 5. 비개발자 1일 과정 운영 가이드

## 5.1 비개발자 과정의 핵심 목표

비개발자 과정의 목표는 코드를 작성하는 것이 아닙니다.  
현업이 알고 있는 업무 지식을 자동화 가능한 요구사항으로 정리하는 것입니다.

최종 산출물:

- 업무 후보 1건
- 업무 시나리오 1페이지
- Skill 초안
- Workflow 초안
- `_workspace` 규칙
- 리스크 레지스터
- 검증 체크리스트
- 다음 액션 플랜

---

## 5.3 세션별 운영 상세

### 5.3.1 업무 후보 선정

강사 목표:

- 수강생이 너무 큰 업무를 고르지 않게 한다.
- 하루 안에 설명 가능한 업무를 선택하게 한다.

좋은 후보 예시:

- 회의록 요약 및 액션 아이템 정리
- 고객 문의 1차 분류
- 주간 보고서 초안 작성
- 문서 검토 전 체크리스트 작성
- 내부 요청서 접수 내용 요약

피해야 할 후보 예시:

- 전사 영업 자동화
- 고객 대응 전체 자동화
- 모든 보고서 자동 작성
- 승인자와 책임자가 불명확한 업무
- 민감정보를 제거할 수 없는 업무

강사 질문:

```text
이 업무는 하루 안에 입력, 처리 과정, 출력, 검토자를 설명할 수 있습니까?
```

```text
이 업무를 처음 보는 사람이 이 설명만 보고 결과물을 상상할 수 있습니까?
```

---

### 5.3.2 Skill 작성 실습

강사 목표:

- Skill을 긴 프롬프트가 아니라 업무 매뉴얼로 작성하게 한다.
- 목적, 입력, 절차, 금지 사항, 출력 형식, 사람 검토 지점을 빠뜨리지 않게 한다.

중간 점검 질문:

```text
이 Skill은 어떤 업무 하나만 다루고 있습니까?
```

```text
입력 자료가 구체적인 파일이나 문서로 설명되어 있습니까?
```

```text
AI가 하지 말아야 할 일이 적혀 있습니까?
```

```text
결과물을 사람이 어떻게 확인해야 하는지 적혀 있습니까?
```

자주 발생하는 문제와 지도 방식:

| 문제 | 강사 대응 |
| --- | --- |
| “잘 정리한다”처럼 표현이 추상적임 | 출력 항목을 구체적으로 쓰게 한다. |
| 입력 자료가 불명확함 | 실제 파일명, 문서명, 출처를 적게 한다. |
| 금지 사항이 없음 | 개인정보, 추정, 승인 전 배포 금지를 질문한다. |
| 사람 검토 지점이 없음 | “누가 최종 책임을 지나요?”라고 묻는다. |

---

### 5.3.3 Workflow 작성 실습

강사 목표:

- 업무 단계와 승인·검토 지점을 시각화하게 한다.
- 예외 흐름과 재작업 경로를 포함하게 한다.

강사 질문:

```text
AI가 초안을 만든 뒤 바로 최종 배포해도 됩니까?
```

```text
정보가 부족하거나 결과가 틀렸을 때 어디로 되돌아가야 합니까?
```

```text
최종 완료는 어떤 상태를 의미합니까?
```

기본 흐름:

```text
요청 접수 → 입력 확인 → AI 초안 생성 → 사람 검토 → 수정/승인 → 최종 저장/배포
```

지도 포인트:

- 사람이 하는 일과 AI가 하는 일을 분리한다.
- 승인 전과 승인 후 산출물 위치를 구분한다.
- 예외 상황을 최소 2개 이상 작성하게 한다.

---

### 5.3.4 Workspace 작성 실습

강사 목표:

- 원본, 중간, 최종, 보관, 임시 파일을 구분하게 한다.
- 파일명 규칙을 너무 복잡하지 않게 만든다.

기본 구조:

```text
_workspace/
  in/        원본 입력 자료
  work/      작업 중 파일, 중간 결과
  out/       최종 산출물
  archive/   완료 후 보관 자료
  tmp/       임시 파일
```

강사 질문:

```text
원본 입력 파일은 수정해도 됩니까?
```

```text
AI가 만든 초안과 승인된 최종본은 어떻게 구분합니까?
```

```text
나중에 문제가 생겼을 때 어떤 파일을 기준으로 재현할 수 있습니까?
```

---

### 5.3.5 리스크와 검증 작성

강사 목표:

- AI 결과물을 그대로 신뢰하지 않고 검증 기준을 만들게 한다.
- 비개발자 수준에서는 체크리스트 중심으로 작성하게 한다.

필수 리스크 유형:

| 리스크 | 확인 질문 |
| --- | --- |
| 누락 | 필요한 항목이 빠지지 않았는가? |
| 왜곡 | 원문과 다른 의미가 되었는가? |
| 과장 | 확인되지 않은 내용을 단정했는가? |
| 개인정보 | 민감정보가 노출되었는가? |
| 승인 누락 | 배포 전 검토자가 확인했는가? |
| 버전 혼선 | 초안과 최종본이 섞였는가? |

강사 질문:

```text
이 결과물이 틀렸을 때 가장 먼저 문제가 되는 부분은 무엇입니까?
```

```text
최종 배포 전에 반드시 확인해야 하는 5가지는 무엇입니까?
```

## 6.3 세션별 운영 상세

### 6.3.1 기준 시나리오 설정

권장 기준 시나리오:

```text
회의록 원문을 입력받아 핵심 요약, 결정사항, 액션 아이템을 작성하고,
사람 검토 후 최종 결과를 workspace/out에 저장한다.
```

강사 포인트:

- 너무 복잡한 실제 시스템 연동을 피한다.
- API 구현보다 문서 계약을 우선한다.
- 비개발자 산출물이 있으면 입력으로 사용한다.

---

### 6.3.2 `SKILL.md` 작성 실습

강사 목표:

- `SKILL.md`를 프롬프트가 아니라 업무 계약으로 작성하게 한다.
- 입력, 출력, 실패 처리, 사람 검토를 반드시 포함하게 한다.

중간 점검 질문:

```text
이 Skill은 하나의 명확한 업무를 다룹니까?
```

```text
출력 구조가 테스트 가능한 형태로 정의되어 있습니까?
```

```text
입력이 부족하거나 모호할 때 어떻게 해야 하는지 정의되어 있습니까?
```

```text
사람 승인 전 최종 산출물로 저장하지 않는 규칙이 있습니까?
```

자주 발생하는 문제와 지도 방식:

| 문제 | 강사 대응 |
| --- | --- |
| `SKILL.md`가 프롬프트처럼 작성됨 | Purpose, Inputs, Output contract, Failure handling 구조로 다시 나누게 한다. |
| 출력 계약이 없음 | Markdown/JSON/표 등 구조를 명시하게 한다. |
| 실패 처리 없음 | missing input, unclear owner, PII detected 등의 경우를 추가하게 한다. |
| 예시가 없음 | 정상 예시 1개와 경계 사례 1개를 요구한다. |

---

### 6.3.3 Orchestrator 작성 실습

강사 목표:

- 여러 단계를 어떻게 연결할지 정의하게 한다.
- context passing과 human gate를 포함하게 한다.

강사 질문:

```text
각 단계가 다음 단계에 넘기는 정보는 무엇입니까?
```

```text
넘기면 안 되는 정보는 무엇입니까?
```

```text
어떤 실패는 재시도하고, 어떤 실패는 중단해야 합니까?
```

```text
사람 승인 전 결과물이 저장될 위치와 승인 후 위치가 다릅니까?
```

지도 포인트:

- 모든 실패를 자동 재시도하지 않게 한다.
- 사람 검토가 필요한 실패를 분리한다.
- Orchestrator가 하위 Skill의 세부 절차까지 중복 작성하지 않도록 한다.

---

### 6.3.4 Task/Sub-agent 설계 실습

강사 목표:

- Task를 입력·출력이 명확한 단위로 나누게 한다.
- Sub-agent는 역할 경계가 분명할 때만 사용하게 한다.

Task 카드 필수 요소:

```text
Task name:
Input:
Output:
Failure modes:
Retry policy:
Idempotency note:
Log keys:
Owner / reviewer:
```

강사 질문:

```text
이 Task는 단독으로 성공 또는 실패를 판단할 수 있습니까?
```

```text
같은 입력으로 두 번 실행해도 결과가 꼬이지 않습니까?
```

```text
이 작업은 별도 Sub-agent가 필요할 만큼 역할이 분리되어 있습니까?
```

금지 패턴:

- Task 이름이 “처리하기”, “정리하기”처럼 모호함
- 입력과 출력이 없음
- 재시도 정책이 모두 동일함
- Sub-agent가 다시 부모에게 같은 작업을 위임함
- 로그 키가 없음

---

### 6.3.5 `AGENTS.md` 작성 실습

강사 목표:

- 저장소 진입점과 읽을 순서를 문서화하게 한다.
- 금지 경로, workspace 규칙, 검증 포인터를 포함하게 한다.

강사 질문:

```text
새로운 개발자나 에이전트가 이 저장소에서 가장 먼저 읽어야 하는 파일은 무엇입니까?
```

```text
절대 수정하면 안 되는 경로는 어디입니까?
```

```text
검증 절차는 어디에 문서화되어 있습니까?
```

```text
보안상 절대 로그에 남기면 안 되는 정보는 무엇입니까?
```

필수 포함 항목:

- Project purpose
- Repository layout
- Read-first order
- Skills
- Workspace rules
- Commands
- Verification
- Security notes
- Prohibited actions

---

### 6.3.6 Smoke Verification 작성 실습

강사 목표:

- 결정적 테스트와 비결정적 스모크 검증을 구분하게 한다.

강사 질문:

```text
정상 입력 1건으로 무엇을 확인할 수 있습니까?
```

```text
반드시 실패하거나 사람 검토로 넘어가야 하는 사례는 무엇입니까?
```

```text
로그에서 성공과 실패를 판단할 수 있는 키는 무엇입니까?
```

필수 항목:

- Input fixture
- Expected output structure
- Required checks
- Failure cases
- Log keys
- Human review criteria

# 9. 자주 나오는 질문과 답변

## Q1. 이 교육은 프롬프트 엔지니어링 교육인가요?

아닙니다. 프롬프트 작성도 일부 포함될 수 있지만, 핵심은 조직이 반복 사용할 수 있는 Skill, Workflow, Workspace, Verification 구조를 만드는 것입니다.

## Q2. 비개발자도 Skill을 작성할 수 있나요?

가능합니다. 비개발자 과정에서 Skill은 업무 매뉴얼입니다. 현업 담당자가 절차, 금지 사항, 검토 기준을 먼저 정리해야 개발자가 구현할 수 있습니다.

## Q3. 개발자 과정에서는 실제 코딩을 하나요?

1일 과정에서는 완성된 앱 구현보다 `SKILL.md`, Orchestrator, Task 설계, `AGENTS.md`, Smoke Verification 같은 하네스 구조 작성에 초점을 둡니다. 코드는 선택 사항입니다.

## Q4. AI가 모든 단계를 자동으로 처리해야 하나요?

아닙니다. 기업 업무에서는 사람 승인과 검토가 들어가는 것이 정상입니다. 자동화의 목표는 사람을 제거하는 것이 아니라 반복 작업을 줄이고 검토 품질을 높이는 것입니다.

## Q5. 검증은 개발자만 하는 것 아닌가요?

아닙니다. 비개발자는 체크리스트와 샘플 입력·기대 출력으로 검증 기준을 만들 수 있습니다. 개발자는 이를 Smoke Test나 CI 검증으로 확장할 수 있습니다.

## Q6. 민감정보가 포함된 업무도 실습할 수 있나요?

실제 민감정보는 사용하지 않는 것이 원칙입니다. 비식별 샘플 데이터 또는 가상 데이터를 사용해야 합니다. 민감정보 처리 기준은 별도 보안 검토가 필요합니다.

## Q7. 오늘 만든 자료가 바로 파일럿으로 이어질 수 있나요?

조직의 준비 수준에 따라 다릅니다. 하지만 오늘 만든 업무 시나리오, Skill 초안, Workflow, Workspace 규칙, 검증 체크리스트는 파일럿 검토의 출발점으로 사용할 수 있습니다.

# 10. 수강생이 어려워하는 포인트와 대응

| 어려움 | 원인 | 강사 대응 |
| --- | --- | --- |
| 업무 범위가 너무 큼 | 자동화 대상을 시스템 전체로 생각함 | 하루 안에 설명 가능한 업무 1건으로 좁히게 한다. |
| Skill이 추상적임 | 프롬프트처럼 작성함 | 입력, 절차, 금지, 출력, 검토 항목으로 나누게 한다. |
| 사람 검토 지점이 빠짐 | AI가 다 해준다고 생각함 | 최종 책임자와 승인자를 묻는다. |
| Workspace를 가볍게 봄 | 파일 관리가 자동화 품질과 연결된다는 인식 부족 | 원본·초안·최종본 혼선 사례를 설명한다. |
| 검증 기준이 없음 | 결과가 좋아 보이면 된다고 생각함 | 누락, 왜곡, 개인정보, 승인 누락 기준을 제시한다. |
| 개발자 Task가 과도하게 분해됨 | 기술적으로 모든 단계를 나누려 함 | I/O와 실패 복구 가치가 있는 단위만 Task로 나누게 한다. |
| AGENTS.md가 README와 중복됨 | 역할 차이를 이해하지 못함 | AGENTS.md는 에이전트와 신규 기여자의 작업 규칙 중심이라고 설명한다. |

# 11. 강사용 중간 점검 체크포인트

## 11.1 오전 종료 전 체크

비개발자 과정:

- [ ] 업무 후보가 하나로 좁혀졌다.
- [ ] 업무 시작 조건이 설명되어 있다.
- [ ] 입력 자료와 최종 산출물이 정의되어 있다.
- [ ] Skill 초안에 목적과 절차가 있다.
- [ ] 금지 사항이 최소 1개 이상 있다.

개발자 과정:

- [ ] 기준 시나리오가 정해졌다.
- [ ] 레포 구조가 만들어졌다.
- [ ] `SKILL.md` 초안이 있다.
- [ ] 입력과 출력 계약이 있다.
- [ ] 실패 처리 항목이 최소 1개 이상 있다.

---

## 11.2 오후 중간 체크

비개발자 과정:

- [ ] Workflow에 사람 검토 지점이 있다.
- [ ] 예외 흐름이 최소 1개 이상 있다.
- [ ] `_workspace` 구조가 작성되었다.
- [ ] 초안과 최종본 저장 위치가 구분된다.

개발자 과정:

- [ ] Orchestrator에 human gate가 있다.
- [ ] Task가 3~5개 정도로 분해되었다.
- [ ] 각 Task에 Input과 Output이 있다.
- [ ] `AGENTS.md`에 read-first order가 있다.
- [ ] 금지 경로 또는 prohibited actions가 있다.

---

## 11.3 종료 전 체크

비개발자 과정:

- [ ] 업무 시나리오 1페이지가 완성되었다.
- [ ] 주요 리스크가 3개 이상 작성되었다.
- [ ] 검증 체크리스트가 5개 이상 작성되었다.
- [ ] 다음 액션이 정리되었다.

개발자 과정:

- [ ] Smoke Verification이 작성되었다.
- [ ] Golden path와 failure case가 있다.
- [ ] Security Notes가 작성되었다.
- [ ] Review Checklist가 작성되었다.
- [ ] 최종 제출 패키지 구조가 정리되었다.
