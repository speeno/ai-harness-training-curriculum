## Slide 1. 비개발자 트랙 · 1일 집중 워크숍

### 화면 문구

**반복 업무를 AI 에이전트가 수행 가능한 Skill, Workflow, Workspace 초안으로 전환하기**

오늘 과정은 코딩 교육이 아닙니다.  
오늘의 목표는 우리 팀의 반복 업무 중 하나를 골라, AI 에이전트가 처리할 수 있을 만큼 명확한 업무 절차로 정리하는 것입니다.

### 발표자 메모

AI에게 일을 맡기기 전에 먼저 사람이 업무를 설명할 수 있어야 합니다. 어떤 입력을 받아야 하는지, 어떤 순서로 처리해야 하는지, 어디에서 사람이 검토해야 하는지, 결과물은 어디에 남겨야 하는지를 정리해야 합니다.

---

## Slide 2. 오늘의 목표

### 화면 문구

오늘 하루가 끝나면 다음 3가지를 가지고 나갑니다.

1. **업무 시나리오 1페이지**  
   - Trigger, Input, Output, Risk 정리

2. **Skill 초안 1건**  
   - 목적, 절차, 금지 사항, 예시 입출력 포함

3. **`_workspace` 폴더 규약 초안**  
   - 입력, 작업 중 파일, 최종 산출물, 임시 파일 위치 정의

### 발표자 메모

오늘의 목표는 완벽한 자동화 시스템을 완성하는 것이 아닙니다. 실제 파일럿 논의로 넘길 수 있는 초안을 만드는 것입니다.

## Slide 4. 자동화에 적합한 업무란?

### 화면 문구

좋은 후보 업무의 조건:

| 기준 | 질문 |
| --- | --- |
| 반복성 | 자주 반복되는 업무인가? |
| 규칙성 | 처리 기준을 문장이나 표로 설명할 수 있는가? |
| 입력·출력 명확성 | 입력 자료와 결과물이 비교적 분명한가? |
| 실수 비용 | 사람이 반복하다가 실수하기 쉬운가? |
| 검토 가능성 | AI 결과를 사람이 확인할 수 있는가? |
| 보안 적합성 | 비식별 샘플로 실습할 수 있는가? |

피해야 할 후보:

- 판단 기준이 전혀 합의되지 않은 업무
- 민감정보를 제거할 수 없는 업무
- 결과 오류 시 즉시 큰 피해가 발생하는 업무
- 승인자나 책임자가 불명확한 업무

### 발표자 메모

가장 좋은 후보는 반복적이고, 규칙이 어느 정도 있으며, 입력과 출력이 문서나 표로 정리될 수 있는 업무입니다.

## Slide 7. Skill = 업무 매뉴얼

### 화면 문구

Skill은 AI에게 전달하는 조직의 업무 매뉴얼입니다.

Skill에 들어갈 내용:

1. 목적
2. 적용 범위
3. 입력 자료
4. 처리 절차
5. 금지 사항
6. 예외 처리
7. 출력 형식
8. 검토 기준

예시:

- 업무명: 고객 문의 요약 및 분류
- 목적: 상담 담당자가 빠르게 내용을 파악하도록 문의 내용을 요약하고 유형을 분류한다.
- 금지 사항: 고객 개인정보를 외부 문서에 그대로 복사하지 않는다.
- 출력: 문의 요약, 유형, 긴급도, 담당 부서 추천

### 발표자 메모

Skill은 단순히 긴 프롬프트가 아닙니다. 사람이 읽어도 업무 절차로 이해할 수 있고, AI가 읽어도 실행 기준으로 사용할 수 있어야 합니다.

---

## Slide 8. 좋은 Skill과 나쁜 Skill

### 화면 문구

| 구분 | 나쁜 Skill | 좋은 Skill |
| --- | --- | --- |
| 목적 | 보고서 잘 만들어줘 | 월간 영업 데이터를 요약해 임원 보고 초안을 만든다 |
| 입력 | 자료 참고 | 매출표, 전월 대비 증감표, 주요 이슈 메모 |
| 절차 | 알아서 정리 | 핵심 수치 추출 → 증감 원인 정리 → 리스크 표시 |
| 금지 | 없음 | 확정되지 않은 수치는 추정으로 표시하지 않는다 |
| 출력 | 보기 좋게 | 제목, 핵심 요약, 표, 리스크, 다음 액션 순서 |
| 검토 | 필요하면 확인 | 팀장 승인 후 배포 가능 |

### 발표자 메모

좋은 Skill은 다른 사람이 읽어도 같은 결과물을 기대할 수 있어야 합니다.

## Slide 10. Orchestrator = Workflow

### 화면 문구

Orchestrator는 업무의 순서와 분기, 승인 지점을 정리한 Workflow입니다.

```text
요청 접수
  ↓
입력 자료 확인
  ↓
AI 초안 생성
  ↓
사람 검토
  ↓
수정 필요?
  ├─ 예 → 재작성 요청
  └─ 아니오 → 최종 저장 / 배포
```

Workflow에 포함할 것:

- 시작 조건
- 처리 단계
- 사람 승인 지점
- 예외 상황
- 재작업 경로
- 최종 완료 조건

### 발표자 메모

기업 업무에서는 중간에 사람이 확인해야 하는 경우가 많습니다. Workflow는 이런 사람 승인과 예외 흐름을 정리하는 도구입니다.

## Slide 12. `_workspace` = 작업 공간 규칙

### 화면 문구

`_workspace`는 AI와 사람이 함께 사용하는 작업 공간 규칙입니다.

```text
_workspace/
  in/        원본 입력 자료
  work/      작업 중 파일, 중간 결과
  out/       최종 산출물
  archive/   완료 후 보관 자료
  tmp/       임시 파일
```

왜 필요한가?

- 원본과 수정본이 섞이지 않게 한다.
- 중간 결과를 추적할 수 있게 한다.
- 최종 산출물을 쉽게 찾을 수 있게 한다.
- 감사와 재현 가능성을 높인다.

### 발표자 메모

입력 파일과 출력 파일이 섞이면 나중에 어떤 자료를 기준으로 결과가 만들어졌는지 알기 어렵습니다.

| 폴더 | 용도 | 파일명 예시 | 보존 규칙 |
| --- | --- | --- | --- |
| `_workspace/in` | 원본 입력 | `2026-05-14_request_original.docx` | 원본 수정 금지 |
| `_workspace/work` | 중간 작업 | `2026-05-14_summary_draft_v0.1.md` | 작업 중 보관 |
| `_workspace/out` | 최종 산출물 | `2026-05-14_summary_final.md` | 승인 후 저장 |
| `_workspace/archive` | 완료 보관 | `2026-05_request_package.zip` | 월별 보관 |
| `_workspace/tmp` | 임시 파일 | `temp_extract.txt` | 작업 후 삭제 |

추가로 정할 것:

- 날짜 형식
- 버전 표기
- 승인 전/후 파일명 구분
- 민감정보 포함 파일 표시 방식

### 발표자 메모

폴더 구조는 복잡할 필요가 없지만 원본, 중간 결과, 최종 결과, 임시 파일은 구분되어야 합니다.

## Slide 14. 리스크와 검증 체크

### 화면 문구

AI 결과물을 그대로 믿지 않기 위한 최소 검증표:

| 리스크 | 확인 질문 | 대응 |
| --- | --- | --- |
| 누락 | 필요한 항목이 빠지지 않았는가? | 필수 항목 체크리스트 |
| 왜곡 | 원문과 다른 의미로 요약하지 않았는가? | 원문 대조 |
| 과장 | 확인되지 않은 내용을 단정하지 않았는가? | 추정/사실 구분 |
| 개인정보 | 민감정보가 노출되지 않았는가? | 마스킹 또는 제거 |
| 승인 누락 | 배포 전 검토자가 확인했는가? | 승인 체크 |

### 발표자 메모

검증은 AI를 불신하기 위한 것이 아니라, 업무에 사용할 수 있을 만큼 책임 있게 만들기 위한 절차입니다.

## Slide 15. 실습 5: 업무 시나리오 1페이지 완성

완료 기준:

- 다른 팀원이 읽어도 이해된다.
- 승인·검토 지점이 보인다.
- 산출물 위치가 보인다.
- 개발자에게 넘길 요구사항으로 사용할 수 있다.

### 발표자 메모

업무 후보, Skill 초안, Workflow, 폴더 규칙, 리스크 체크를 하나의 시나리오 문서로 합칩니다.

## Slide 17. 발표 및 피드백 기준

### 화면 문구

| 평가 항목 | 확인 질문 |
| --- | --- |
| 업무 명확성 | 어떤 업무인지 한 문장으로 설명되는가? |
| 자동화 적합성 | 반복성, 규칙성, 입력·출력이 보이는가? |
| Skill 품질 | 절차, 금지 사항, 출력 형식이 있는가? |
| Workflow 품질 | 승인자와 예외 경로가 보이는가? |
| Workspace 품질 | 원본·중간·최종 파일 위치가 구분되는가? |
| 리스크 인식 | 오류, 누락, 보안 위험을 식별했는가? |
| 다음 단계 | 파일럿 또는 개발자 트랙으로 넘길 수 있는가? |

피드백 문장 예시:

- “이 업무의 입력 자료가 조금 더 명확하면 좋겠습니다.”
- “사람 검토 지점이 한 단계 더 필요해 보입니다.”
- “최종 산출물의 저장 위치가 명확해서 좋습니다.”
- “개인정보 처리 기준을 추가하면 더 안전할 것 같습니다.”

### 발표자 메모

피드백은 평가가 아니라 개선을 위한 질문이어야 합니다.

## Slide 19. 다음 단계

### 화면 문구

1일 과정 이후 가능한 경로:

1. **팀 내부 검토**
   - 업무 담당자, 승인자, 보안 담당자와 초안 검토

2. **비개발자 3일 과정 확장**
   - 후보 업무 3~5건 우선순위화
   - 리스크 레지스터와 Workflow 고도화

   - Skill 초안을 `SKILL.md`로 전환
   - Workflow를 Orchestrator 설계로 전환
   - 검증 체크리스트를 스모크 테스트로 전환

4. **파일럿 실행 준비**
   - 샘플 데이터 확정
   - 승인자 지정
   - 성공 기준과 중단 조건 정의

오늘 만든 자료는 끝이 아니라 시작점입니다. 팀 내부 검토 후 3일 과정으로 확장하거나 개발자 트랙으로 넘겨 구현 가능성을 검토할 수 있습니다.

# 4. 강사용 진행 팁

## 오전 운영 팁

- 개념 설명은 짧게 유지한다.
- 비개발자에게 기술 용어를 직접 설명하기보다 업무 비유를 사용한다.
- “Skill = 업무 매뉴얼”, “Orchestrator = Workflow”, “Workspace = 폴더 규칙”이라는 표현을 반복한다.
- 자동화 후보 선정에서 너무 큰 업무를 고르면 반드시 좁혀준다.

## 오후 운영 팁

- 오후에는 반드시 손으로 작성하게 한다.
- 완성도보다 빈칸을 줄이는 것을 우선한다.
- 발표 전에는 각 팀이 최소 1개 리스크와 1개 사람 검토 지점을 포함했는지 확인한다.
- 피드백은 비판보다 질문 중심으로 유도한다.
